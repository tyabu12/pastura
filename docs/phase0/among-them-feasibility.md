# Among Them — 技術的フィージビリティ分析

## 1. ベンチマーク実測値（公式 LiteRT-LM ベンチマーク）

条件: 1024 prefill tokens / 256 decode tokens / context 2048

### Gemma 4 E2B（2.58 GB）

| デバイス | Backend | Prefill | **Decode (tok/s)** | TTFT | メモリ |
|---|---|---|---|---|---|
| iPhone 17 Pro | CPU | 532 | **25.0** | 1.9s | 607 MB |
| iPhone 17 Pro | GPU | 2,878 | **56.5** | 0.3s | 1,450 MB |

### Gemma 4 E4B（3.65 GB）

| デバイス | Backend | Prefill | **Decode (tok/s)** | TTFT | メモリ |
|---|---|---|---|---|---|
| iPhone 17 Pro | CPU | 159 | **9.7** | 6.5s | 961 MB |
| iPhone 17 Pro | GPU | 1,189 | **25.1** | 0.9s | 3,380 MB |

> ベンチマークは iPhone 17 Pro 基準。iPhone 16e は RAM 8GB なのでメモリ制約がより厳しい。

## 2. シミュレーション速度の試算

### 前提条件

- エージェント1回の応答 ≈ 100〜200 tokens
- 5体のエージェント × 1ターン = 5回の推論
- LLMは1つを時分割（並列不可）

### E2B GPU モードの場合（56.5 tok/s）

| 項目 | 計算 |
|---|---|
| 1回の推論（150 tok） | 約 2.7 秒 |
| 1ターン（5体） | 約 13.5 秒 |
| 10ターン | 約 2.3 分 |
| 100ターン | 約 23 分 |
| 1,000ターン | 約 3.8 時間 |

### E2B CPU モードの場合（25.0 tok/s）— バックグラウンド想定

| 項目 | 計算 |
|---|---|
| 1回の推論（150 tok） | 約 6.0 秒 |
| 1ターン（5体） | 約 30 秒 |
| 100ターン | 約 50 分 |
| 1,000ターン | 約 8.3 時間 |

### E4B GPU モードの場合（25.1 tok/s）

| 項目 | 計算 |
|---|---|
| 1回の推論（150 tok） | 約 6.0 秒 |
| 1ターン（5体） | 約 30 秒 |
| 100ターン | 約 50 分 |

> **結論**: E2B で GPU なら「1分で4〜5ターン」。放置ゲーとしては十分な速度。
> E4B は品質は上がるが速度が半減し、メモリ 3.4GB 消費は iPhone 16e（RAM 8GB）では厳しい。

## 3. 最大のリスク：iOS バックグラウンド実行制約

### 現状の iOS（〜iOS 18）

iOSのバックグラウンド実行はAppleが厳しく制限しており、**Among Them のコア体験「放置して勝手に進行」に直接影響する**。

| API | 実行可能時間 | 条件 |
|---|---|---|
| `beginBackgroundTask` | 約30秒 | アプリがバックグラウンドに入った直後のみ |
| `BGAppRefreshTask` | 約30秒 | システム裁量。1日数回程度 |
| `BGProcessingTask` | 数分程度 | 充電中 + Wi-Fi 接続時に優先。保証なし |

→ **「寝てる間に1000ターン進む」は iOS 18 以前では実現困難**。

### iOS 26（2025 WWDC）の新 API

`BGContinuedProcessingTask` が導入された。

- ユーザーが明示的にアクションを起こして開始（ボタンタップ等）
- システムUIで進捗が表示される
- バックグラウンドでもGPUアクセス可能（対応デバイスのみ）
- ユーザーがいつでもキャンセル可能

→ **これは Among Them にとって非常に有利な変化**。ただしユーザー起点の操作が必須で「完全放置」とは若干異なるUXになる。

### 現実的な設計方針

| 状況 | 挙動 |
|---|---|
| フォアグラウンド | GPU フル速度で推論（56.5 tok/s） |
| バックグラウンド（iOS 26+）| `BGContinuedProcessingTask` で継続。進捗UI表示 |
| バックグラウンド（iOS 18以前）| `BGProcessingTask` で充電中のみ断続実行 |
| アプリ復帰時 | 中断地点から再開。「○○ターン進みました」で通知 |

## 4. メモリ制約の分析

### iPhone 16e（RAM 8GB）でのメモリバジェット

| 用途 | 推定消費 |
|---|---|
| OS + バックグラウンドアプリ | 約 3〜4 GB |
| アプリ自体（UI, DB等） | 約 200〜300 MB |
| **LLM に使える残り** | **約 3.5〜4.5 GB** |

| モデル | メモリ消費 | 判定 |
|---|---|---|
| E2B GPU | 1,450 MB | ✅ 余裕あり |
| E2B CPU | 607 MB | ✅ 非常に軽い |
| E4B GPU | 3,380 MB | ⚠️ ギリギリ〜NG |
| E4B CPU | 961 MB | ✅ 可能 |

> **結論**: iPhone 16e（8GB RAM）では **E2B 一択**。
> E4B は iPhone 17 Pro（12GB 想定）以上でのオプション提供が妥当。

## 5. コンテキスト管理とペルソナ演じ分けの課題

### コンテキストウィンドウ

- E2B / E4B ともに最大 128K tokens
- ただし LiteRT-LM 経由だとベンチマークは 2K context、最大 32K まで

### エージェントあたりのプロンプト構造（推定）

```
[システムプロンプト]           ≈ 200 tok
  - シナリオ設定
  - このエージェントのペルソナ
  - 行動ルール

[記憶の要約]                  ≈ 200〜500 tok
  - 過去の重要な出来事の圧縮版

[直近の会話ログ]              ≈ 500〜1500 tok
  - 最新5〜10ターンの全文

[行動指示]                    ≈ 50 tok
  - 「あなたの次の行動と発言を決めてください」

合計: 約 950〜2,250 tokens/回
```

### 2Bモデルでペルソナ演じ分けは可能か？

**ここが最大の品質リスク**。

- 良い点: Gemma 4 は function calling 対応で構造化出力が得意。行動を JSON で返させれば安定
- 懸念点: E2B クラスだとキャラの「口調」「判断の癖」が似通る可能性が高い
- 対策案:
  1. ペルソナをシステムプロンプトで強く制約（口調の具体例を複数提示）
  2. 応答フォーマットを構造化（JSON: action, speech, inner_thought）
  3. temperature / top-p のエージェントごとの微調整
  4. 「キャラが似通っても面白い」シナリオを選ぶ（人狼・裁判など役割が明確なもの）

## 6. 発熱・バッテリーの現実

### GPU 推論時の消費電力

- GPU フルで E2B を回すと、iPhone はかなり発熱する（実体験とも一致するはず）
- 連続 1 時間以上の GPU 推論はサーマルスロットリングで速度が落ちる可能性大

### 対策設計

| モード | 推論ペース | 想定用途 |
|---|---|---|
| 高速モード | GPU フル、毎ターン即座 | フォアグラウンドで観察中 |
| 省電力モード | CPU、5〜10秒間隔 | バックグラウンド放置 |
| 充電モード | GPU フル、休憩なし | 充電中に大量ターン |
| スリープモード | 推論停止 | バッテリー残量20%以下 |

## 7. 総合判定

### Go / No-Go 判定

| 項目 | 判定 | 備考 |
|---|---|---|
| LLM のスマホ動作 | ✅ Go | E2B は iPhone 16e でも動く |
| 推論速度 | ✅ Go | GPU 56 tok/s で十分実用的 |
| メモリ | ✅ Go（E2B） | E4B は上位端末のみ |
| バックグラウンド実行 | ⚠️ 条件付きGo | iOS 26 の新 API で大幅改善。iOS 18以前は制限あり |
| ペルソナ品質 | ⚠️ 要検証 | E2B でキャラ演じ分けの実験が最優先 |
| 発熱・バッテリー | ⚠️ 要設計 | 推論ペース制御で対処可能 |

### 次にやるべきこと（優先順）

1. **ペルソナ演じ分け実験**（最重要）
   - Gemma 4 E2B で 3〜5 キャラを設定し、10ターン程度の会話を実際に回す
   - キャラの区別がつくか、会話が面白いかを主観評価
   - プロンプト構造を何パターンか試す

2. **バックグラウンド動作の PoC**
   - iOS 26 beta で `BGContinuedProcessingTask` + LiteRT-LM の組み合わせ検証
   - GPU がバックグラウンドで使えるか確認

3. **収束問題の検証**
   - 50ターン、100ターンまで回して会話がループしないか確認
   - 記憶の要約→再注入パイプラインのプロトタイプ
