# ADR-002: llama.cpp as Interim On-Device LLM Backend

> **Status:** Accepted
> **Date:** 2026-04-08
> **Context:** Phase 1 MVP requires on-device LLM inference for TestFlight distribution.
> LiteRT-LM Swift SDK is not yet released; its C API lacks iOS GPU support.

## Summary

Use llama.cpp (via Metal GPU) as the interim production LLM backend for TestFlight.
Migrate to LiteRT-LM when the official Swift SDK ships with iOS GPU support.

---

## 1. Why Not LiteRT-LM Today?

### LiteRT-LM Status (as of 2026-04-08)

| SDK | Status |
|-----|--------|
| Python | Stable |
| Kotlin (Android) | Stable |
| C++ | Stable |
| **Swift** | **In Dev ("Coming Soon")** |

- Swift SDK: no SwiftPM package, no documentation, no release timeline.
- C API (`c/engine.h`): works on iOS but **GPU inference is unavailable** on iOS.
  iOS prebuilts are missing entirely ([LiteRT-LM Issue #1050](https://github.com/google-ai-edge/LiteRT-LM/issues/1050)).
  CPU-only mode yields ~9.7 tok/s for E4B (the larger 4B variant) on iPhone — usable but suboptimal.
- AI Edge Gallery iOS app exists on App Store (uses LiteRT-LM C/C++ API internally)
  but its source code is not public ([gallery Issue #420](https://github.com/google-ai-edge/gallery/issues/420)).

Reference: [Zenn article on LiteRT-LM iOS integration](https://zenn.dev/yoshitetsu/articles/ac05dc4a71650d)
documents the C API build process (Bazel, 609 static libs, 360 MB unified archive, `-force_load`,
16 MB stack allocation, dylib-to-framework conversion).

### Decision

LiteRT-LM's C API path is viable but high-effort with no GPU payoff on iOS today.
llama.cpp offers Metal GPU support, mature iOS tooling, and Gemma 4 day-one support.
Use llama.cpp as the interim backend.

## 2. SwiftPM Package Selection

### Evaluation Criteria

- iOS platform support (`.iOS(.v16)` or later)
- Metal GPU shader compilation for iOS
- Swift 6.x strict concurrency compatibility
- Upstream llama.cpp tracking (Gemma 4 requires recent builds)
- Active maintenance

### Candidates Evaluated

| Package | iOS | Swift | Upstream | Last Updated | Verdict |
|---------|-----|-------|----------|-------------|---------|
| **mattt/llama.swift** | `.iOS(.v16)` | 6.0 | b8694 (daily) | 2026-04-07 | **Selected** |
| Official xcframework | N/A (manual) | N/A | Current | Daily releases | Fallback |
| tattn/LocalLLMClient | `.iOS(.v17)` | 6.1 | b6871 (~1800 behind) | 2026-01-31 | Too stale |
| eastriverlee/LLM.swift | `.iOS(.v16)` | 5.9 | Unknown | 2025-12-06 | Too stale |
| StanfordBDHG/llama.cpp | `.iOS(.v17)` | 5.9 | 2+ years stale | 2024-05-12 | Dead |

### Decision

**Primary: [mattt/llama.swift](https://github.com/mattt/llama.swift)**

- Thin SPM wrapper that re-exports the official llama.cpp xcframework from each release
- Tracks upstream within hours — Gemma 4 support guaranteed
- Zero extra dependencies — just the C API, which is exactly what `LlamaCppService` needs
- Created by Mattt (NSHipster), established Swift community member

**Fallback: Official xcframework direct consumption**

If mattt/llama.swift becomes unmaintained, download the xcframework directly from
[llama.cpp releases](https://github.com/ggml-org/llama.cpp/releases) and add as a
local binary SPM target. This is what mattt/llama.swift automates.

### Pin Strategy

Pin to a **specific release tag** (not `main` branch) to ensure reproducible builds.
PR1 should select the latest tag at implementation time and verify Gemma 4 E2B stability:
model loads without crash, JSON output parses successfully, no hangs over 10+ inferences.
Document the chosen tag in the PR description.

## 3. Model Specification

### Gemma 4 E2B

| Property | Value |
|----------|-------|
| Effective parameters | 2B (active during inference) |
| Total parameters | 5.1B (Per-Layer Embeddings) |
| Quantization | Q4_K_M |
| Download size | ~3.1 GB |
| Runtime RAM | ~5 GB (weights + KV cache + Metal buffers) |
| Context limit | 8192 tokens (configured, not model max) |
| GGUF source | [unsloth/gemma-4-E2B-it-GGUF](https://huggingface.co/unsloth/gemma-4-E2B-it-GGUF) (Q4_K_M) |
| License | Apache 2.0 |

### Why Q4_K_M?

- ~4.5 bits/param average — best quality/size balance for on-device
- ~3.1 GB download is practical for mobile (< App Store 4 GB OTA limit)
- ~5 GB RAM fits 8 GB devices, but headroom is tight (~0.5-1.5 GB after iOS)
- Memory warnings should be expected; `didReceiveMemoryWarning` handling is critical
- Q2_K (~1.5 GB) has significant quality loss; Q8_0 (~5.4 GB) wastes bandwidth/RAM

### Why 8192 Context Limit?

- Pastura simulations rarely exceed 8K tokens of accumulated context
- Bounds KV cache memory (~200 MB at Q8 KV for 8K vs ~800 MB for 32K)
- Bounds prefill time (~20-40s for 8K vs potentially minutes for 32K)
- Can be relaxed later based on real-device profiling

### Model Fallback Chain

If Gemma 4 E2B GGUF proves unstable on llama.cpp (support is <1 week old),
fall back to an older, well-tested model:

1. **Gemma 3 4B Q4_K_M** (~2.5 GB, well-established llama.cpp support)
2. **Gemma 2 2B Q4_K_M** (~1.5 GB, oldest and most stable)

## 5. Performance Expectations

### Inference Speed (estimated, llama.cpp + Metal GPU)

| Metric | iPhone 15 Pro (A17) | iPhone 16 Pro (A18) |
|--------|--------------------|--------------------|
| Decode | ~20-35 tok/s | ~25-40 tok/s |
| Prefill (4K tokens) | ~200-400 tok/s | ~250-500 tok/s |

### Comparison with LiteRT-LM (reference)

| Backend | Decode (E2B, GPU) | Notes |
|---------|-------------------|-------|
| LiteRT-LM | ~56.5 tok/s | iPhone 17 Pro benchmark (newer HW) |
| **llama.cpp** | **~20-40 tok/s** | **iPhone 15/16 Pro est.; gap may narrow on same HW** |

### Simulation Time Estimates

For a typical 50-inference simulation (~200 tokens output each, ~2K avg prompt):

| Phase | Time |
|-------|------|
| Prefill (cumulative) | ~3-5 min |
| Decode (cumulative) | ~4-8 min |
| Thermal throttling (20-40% slowdown after 2-5 min) | +2-4 min |
| **Total estimate** | **~10-20 min** |

ADR-001 §8 quotes "50 inferences ≈ 2.3 min" based on LiteRT-LM decode-only math.
Real-world llama.cpp performance is 4-8x slower due to: (a) lower decode speed,
(b) prefill time not previously accounted for, (c) thermal throttling on iPhone.

### Thermal Throttling Mitigation

- Monitor `ProcessInfo.processInfo.thermalState`
- At `.serious` or `.critical`: insert 200ms pause between inferences
- Log thermal state transitions for TestFlight diagnostics

## 6. Architecture: LlamaCppService

### Layer Placement

```
LLM/
├── LLMService.swift          (protocol — unchanged)
├── OllamaService.swift       (dev — unchanged)
├── MockLLMService.swift      (test — unchanged)
├── LlamaCppService.swift     (NEW — llama.cpp backend)
├── JSONResponseParser.swift  (unchanged)
└── LLMError.swift            (unchanged)

App/
├── ModelManager.swift         (NEW — model download/management)
├── AppDependencies.swift      (UPDATED — deferred LLMService init)
└── ...
```

### Why ModelManager in App/, not LLM/?

`LLM/` depends on `Models` only (CLAUDE.md dependency rules). `ModelManager` handles:
- HTTP download (URLSession)
- File system management (Documents directory)
- Device capability checking (ProcessInfo)
- Download progress reporting (for UI)

These are App-layer concerns. `LlamaCppService` receives a model file path
via its constructor (`init(modelPath: String)`), mirroring how `OllamaService`
receives `baseURL`.

### Concurrency Contract

Matching OllamaService (ADR-001 §7):

- `unloadModel()` MUST NOT be called concurrently with `generate()`.
- Engine executes inferences sequentially — no concurrent `generate()` calls.
- Memory warning handler (`didReceiveMemoryWarning`) must cancel the running
  simulation first, then call `unloadModel()`.
- Thread safety for `isModelLoaded` state: `OSAllocatedUnfairLock<Bool>`.

### Chat Template and Output Format

**Chat template**: Use llama.cpp's built-in chat template application for Gemma 4.
llama.cpp reads the template from the GGUF metadata and applies it automatically
via `llama_chat_apply_template()`. `LlamaCppService.generate(system:user:)` passes
the system and user strings; llama.cpp formats them into Gemma's
`<start_of_turn>user\n...<end_of_turn>` format internally.

**JSON output**: Continue using the existing retry-based `JSONResponseParser` approach
(same as OllamaService). Do NOT use GBNF grammar constraints for MVP — the parser
already handles Gemma's thinking tags and noisy output reliably. GBNF grammar may be
explored in Phase 2 if JSON parse failure rates are high in TestFlight telemetry.

### Sampling Parameters

Match OllamaService defaults:
- Temperature: 0.8
- Max tokens: 1000
- Top-k and top-p: use llama.cpp defaults (top-k=40, top-p=0.95)
- Repeat penalty: use llama.cpp default (1.1)

### AppDependencies Deferred Initialization

Current `AppDependencies` creates `llmService` eagerly in `init()` as a `let` property.
For `LlamaCppService`, creation must wait until the model file is on disk.

Solution: use the existing `PasturaApp.dependencies: AppDependencies?` pattern.
- `nil`: show model download screen
- After download: create `AppDependencies` with `LlamaCppService(modelPath:)`
- `let llmService` stays immutable — no change to Sendable reasoning

## 7. App Lifecycle

### Background/Foreground Transitions

When iOS suspends the app, Metal GPU buffers may be invalidated by the OS.
Resuming with stale buffers causes `EXC_BAD_ACCESS` or corrupt output.

| Event | Action |
|-------|--------|
| `scenePhase == .background` | Pause simulation (existing `isPaused` flag) |
| `willResignActive` | Do NOT unload model (would lose KV cache state) |
| Return to foreground | Validate Metal context; if invalid, unload + reload model |
| `didReceiveMemoryWarning` | Cancel simulation, then `unloadModel()` |

### Checkpoint Safety

Consider saving simulation checkpoint state before entering background,
so a jetsam kill during suspension doesn't lose progress. (May defer to Phase 2
if complexity is high.)

## 8. Migration Path to LiteRT-LM

When LiteRT-LM Swift SDK ships with iOS GPU support:

1. Replace `LlamaCppService.swift` with `LiteRTLMService.swift`
2. Switch model format from GGUF to `.litertlm`
3. Update `ModelManager` download URL
4. No changes to Engine, Views, or App layer (protocol abstraction)

The `LLMService` protocol was specifically designed for this swap (ADR-001 §7).

### Monitoring

Track these for migration readiness:
- [LiteRT-LM releases](https://github.com/google-ai-edge/LiteRT-LM/releases) — watch for Swift SDK
- [LiteRT-LM Issue #1050](https://github.com/google-ai-edge/LiteRT-LM/issues/1050) — iOS GPU prebuilts
- [Gallery Issue #420](https://github.com/google-ai-edge/gallery/issues/420) — iOS source code

## 9. Risks and Mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| llama.cpp Gemma 4 support unstable (<1 week old) | High | Pin to tested release; integration tests; model fallback chain |
| mattt/llama.swift becomes unmaintained | Medium | Fallback to official xcframework direct consumption |
| 8 GB RAM insufficient for Q4_K_M + app | Medium | Runtime memory check; consider Q3_K_M if needed |
| Thermal throttling degrades UX | Medium | Inter-inference pauses; thermalState monitoring |
| Metal buffers invalid after background | High | Validate on foreground; reload if needed |
| GGUF model hosting dependency | Low | Use HuggingFace (free, community standard); URL configurable |
| CI Metal shader build | Medium | Verify llama.cpp xcframework compiles on GitHub Actions macOS runners; may need specific Xcode version |
| App binary size increase | Low | llama.cpp xcframework adds ~20-50 MB; well within TestFlight/App Store limits |
| Gemma license compliance | Medium | Gemma Terms of Use require acceptance; may need in-app terms display before model download |
