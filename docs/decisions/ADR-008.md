# ADR-008: Route Identity vs Render-Time Hints

> **Status:** Accepted
> **Date:** 2026-04-26
> **Context:** Phase 2. After PR #249 items 1–5 fixed view-body flash
> via deferred `@State viewModel` assignment, residual title-bar
> pop-in remained: `.navigationTitle` is preference-propagated and
> updates after `.task`-triggered loads complete, so a `.large` title
> bar slides in empty for ~30–80ms before the scenario name appears.
> Operational rule lives in `.claude/rules/navigation.md`
> § "Render-time hints — RouteHint".

---

## Decision

Introduce `RouteHint<T: Hashable & Sendable>` — a value-type wrapper
whose `==` is always `true` and whose `hash(into:)` is a no-op — and
embed it as the type of any `Route` enum associated value that carries
**render-time hints** (placeholders, animation params) rather than
**navigation identity**. The wrapped value is read normally via
`.value`; only `==` and `hashValue` are blind to it. The first
consumers are `Route.scenarioDetail.initialName` and
`Route.simulation.initialName`, both `RouteHint<String>`.

Title binding becomes a 3-tier fallback: loaded view-model name →
push-time `initialName` hint → empty defensive default.

## Why this shape

`Route` already conforms to `Hashable` for `NavigationStack(path:)`
diffing and `AppRouter.pushIfOnTop(expected:next:)` guards. Adding a
plain `String?` `initialName` to a case would make
`Route.scenarioDetail(scenarioId: "x")` (default-nil) and
`Route.scenarioDetail(scenarioId: "x", initialName: "Foo")` compare
unequal under auto-synthesized `Hashable`. Future `pushIfOnTop`
callers that omit the hint when constructing `expected` would silently
fail their guards — a class of bug PR #93 already paid for once.

`RouteHint`'s identity-neutral `==` lets `Route`'s auto-synthesized
`Hashable` keep working unchanged: `scenarioId` continues to drive
identity, the hint participates only in display.

## Alternatives Considered

| | Approach | Pro | Con | Verdict |
|---|---|---|---|---|
| α | Manual `==` / `hash(into:)` on `Route` ignoring the hint | Familiar Swift idiom | Boilerplate scales with `Route` case count; switch update easy to forget when adding cases | Rejected — maintenance cost grows with future `Route` additions |
| **β** | **`RouteHint<T>` identity-neutral wrapper** | One-time type definition; future routes reuse the pattern; intent encoded in the type system | Pastura-specific idiom (~10 lines + doc); slight call-site verbosity (`.init(name)`) | **Accepted** |
| γ | Cache layer in `AppDependencies` keyed by scenario id | `Route` stays untouched; Apple HIG-faithful | Responsibility scattered across cache holder / populator / invalidator; mutable state needs Sendable design; cache invalidation surface | Rejected — sub-system overhead disproportionate to the problem |

Detailed evaluation in PR #249 discussion (responsibility split,
testability, type safety, code discoverability, Phase 3 scalability,
Apple HIG alignment, Sendable, NavigationStack semantics, state
restoration, idiom familiarity).

## Consequences

- **Pro: identity-neutral by construction.** `pushIfOnTop` callers
  may omit the hint when constructing `expected`; the guard still
  matches as long as `scenarioId` (the identity-bearing field) does.
  Pinned by `AppRouterTests.pushIfOnTopMatchesIdenticalScenarioIdRegardlessOfHint`
  and `pushIfOnTopFailsWhenScenarioIdDiffers`.
- **Pro: future-additive.** New `Route` cases that need a render-time
  hint declare it as `RouteHint<T>` instead of plain `T`; auto-
  synthesized `Hashable` continues to use only identity-bearing
  fields. No per-case `==` boilerplate.
- **Con: documentation load-bearing.** A reviewer expecting
  Equatable-equal-implies-interchangeable will be surprised that
  `RouteHint<String>("Foo") == RouteHint<String>("Bar")` while
  `.value` differs. The type's header doc-comment carries the
  warning explicitly.
- **Con: nonisolated requirement.** Under
  `SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor`, the custom `==` /
  `hash(into:)` would otherwise inherit MainActor and fail the
  nonisolated `Hashable` conformance. `RouteHint` is therefore
  declared `nonisolated` at the type level. `Route` itself remains
  unannotated because its auto-synthesized `Hashable` does not have
  this issue.
- **Access modifier:** `RouteHint` is `internal`, matching `Route`'s
  current access level. If `Route` is ever promoted to `public` for
  SPM extraction of `App/`, `RouteHint` must follow.
- **State restoration (future):** Adding `Codable` to `Route` would
  require choosing whether to encode `RouteHint`'s `.value`. Encoding
  it preserves the title across app-kill / restoration; not encoding
  it surfaces the original placeholder briefly on restore. Either is
  defensible; revisit if state restoration is added.

## Amendment 2026-04-29 — Sim sink pivots from `.navigationTitle` to `GameHeader`

### Context

#297 PR 3 (header content unification — Variant C, see PR description
and design hand-off `HEADER_UPDATE.md`) replaced the Sim screen's
single-line `Round X/Y` `PhaseHeader` with the new two-row `GameHeader`
component. `GameHeader` row 1 carries the scenario name as its primary
title — which means surfacing the same name in Sim's nav bar
(`.navigationTitle(scenario?.name ?? initialName ?? "")`) would be a
visible duplication, defeating the unification goal.

### Pivot

`Route.simulation.initialName: RouteHint<String>` now feeds
`GameHeader.initialName` instead of `.navigationTitle()`. The 3-tier
fallback chain is unchanged in shape:

1. `scenario?.name` — authoritative once `loadAndRun()` resolves.
2. `initialName` — push-time hint (this ADR's payload).
3. Empty string — defensive default.

But the consumer rendering it has shifted from the iOS nav bar to
the GameHeader's row 1 title slot. The "first-frame pop-in" problem
this ADR was written to prevent is preserved verbatim: without
`initialName`, the GameHeader would show an empty title for the
duration between push and `loadAndRun()` completion (typically
sub-second on warm cache, multi-second on cold parse), creating the
same jarring late-fill the original nav-title path had.

`SimulationView`'s nav bar is **hidden entirely** via
`.toolbar(.hidden, for: .navigationBar)` so the GameHeader can
extend into the top safe area (`extendsIntoTopSafeArea: true`),
matching Demo's chrome exactly. Real-device testing showed even an
empty `.navigationTitle("")` left ~44pt of unused vertical space
above the header, making the chat-stream uncomfortably cramped on
smaller iPhones — a symptom that did not appear on simulator
testing but was load-bearing on hardware. Back navigation goes
through the swipe-back gesture (preserved by iOS even when the nav
bar is hidden inside a `NavigationStack`) and VoiceOver's 2-finger
Z-gesture for accessibility-first navigation.

### Scope of pivot

`Route.scenarioDetail.initialName` is **unchanged** — ScenarioDetail
still uses `.navigationTitle()` because it has no equivalent
`GameHeader` (the chat-stream lives behind that screen, not on it).
The two `RouteHint<String>` consumers now have divergent rendering
sinks but identical semantics (push-time first-frame placeholder).

### What stays from the original decision

- The `RouteHint<T>` wrapper itself — its identity-neutral
  `Hashable` is what enables `pushIfOnTop(expected:next:)` to compare
  routes correctly across hint values, and that property is unchanged.
- Identity-neutrality warning in the type's header doc-comment — the
  pivot does not change the comparison contract.
- `internal` access modifier — pivot is internal to `App/`.
- `nonisolated` declaration — same MainActor-isolation reasoning.

### What was considered and rejected

- **Keeping `.navigationTitle(scenario?.name ?? initialName ?? "")`
  AND showing the GameHeader** (accept the duplication). Lowest-risk
  option — preserves back-button label, VoiceOver announcement, and
  Handoff metadata without any code change beyond the GameHeader
  insertion. Rejected on intent: visible duplication of the scenario
  name on the same screen defeats the unification goal of #297 PR 3
  (the header unification was explicitly motivated by Demo → Sim
  transition feeling fragmented; carrying the same name in two chrome
  surfaces makes the screen busier, not more unified).
- **Empty `.navigationTitle("")` with the nav bar still visible.**
  Initial pivot considered this option for back-chevron preservation
  (iOS uses the previous view's title for the back-button label, so
  the chevron would still read "ScenarioDetail" even with the
  current view's title empty). Rejected after **real-device testing**
  showed the empty nav bar wasted ~44pt of vertical space above the
  GameHeader, making the chat-stream uncomfortably cramped on
  smaller iPhones. The swipe-back gesture is the standard iOS
  navigation primitive and remains active when the nav bar is
  hidden inside a `NavigationStack`; VoiceOver gets the 2-finger
  Z-gesture for accessibility. The visible chevron's
  discoverability cost is real but bounded — Pastura's audience is
  iOS-native users for whom swipe-back is muscle memory. Demo
  already operates without a back chevron (presented as
  `.fullScreenCover`); the unification goal benefits from Sim
  matching that chrome.
- **Removing `Route.simulation.initialName` entirely**. Tempting given
  Sim's nav title is gone — the hint serves only the GameHeader.
  But the hint's role (first-frame placeholder for an async-loaded
  identifier) is identical regardless of which sink renders it; the
  consumer's identity changes, the producer's contract does not.
  Removing would re-introduce the pop-in flicker the original ADR
  blocked.
