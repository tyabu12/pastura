# ADR-003: Background Simulation Continuation via iOS 26 BGContinuedProcessingTask

> **Status:** Accepted (revised 2026-04-16 for suspend/resume mechanism)
> **Date:** 2026-04-14 (initial) / 2026-04-16 (§4, §5 rewrite + §10)
> **Context:** Phase 2 — ADR-002 §7 documented that backgrounding the app pauses the
> simulation. iOS 26 introduces `BGContinuedProcessingTask`, enabling CPU work to continue
> while the app is backgrounded.

## Revision Note (2026-04-16)

Real-device testing after the initial ADR revealed that iOS denies Metal GPU work
from background within milliseconds of `scenePhase = .background` — much faster than
the original phase-boundary `checkPaused` mechanism could respond. An in-flight
`llama_decode` consistently failed with Metal error -3 before the pause checkpoint
fired, breaking mid-run.

The design was reworked around a cooperative `SuspendController` owned by the VM and
attached to the LLM. §4 (pattern) and §5 (state machine) were rewritten. The original
`pauseAndAwaitConfirmation` / `.simulationPaused` flow was removed. A new §10
authoritatively lists the suspend/resume contract each layer upholds. The
`checkPaused` call site between phases is kept, but only for the user-initiated
pause button — scene-phase handling no longer touches `isPaused`.

## Summary

Add opt-in background simulation continuation for iOS 26+ using `BGContinuedProcessingTask`.
When the user enables the toggle in SimulationView and backgrounds the app, the model
is reloaded from GPU to CPU mode so inference can continue under Background Task API limits.
The feature is gated on iOS 26+ availability and `LlamaCppService` (the only backend that
supports hot reload). In-flight inference is interrupted cooperatively via a
`SuspendController` so iOS does not deny Metal GPU work mid-decode.

---

## 1. Context: What Changed and Why Now

### ADR-002 §7 (Original Lifecycle)

ADR-002 §7 established that `scenePhase == .background` triggers a pause — Metal GPU buffers
may be invalidated by the OS when the app is suspended, and resuming with stale buffers
causes `EXC_BAD_ACCESS` or corrupt output. Background inference was not attempted.

### iOS 26: BGContinuedProcessingTask

iOS 26 introduces `BGContinuedProcessingTask` (part of BackgroundTasks framework) — a
user-initiated background task type designed for long CPU-bound work that the user started
and expects to continue while the app is backgrounded (e.g., large file processing, AI inference).

Key properties:
- **User-initiated requirement**: the system requires that the task is triggered directly by
  a user action, not automatically. A toggle button tap in SimulationView satisfies this.
- **CPU work only (iPhone)**: background GPU access is iPad-only and requires a separate
  entitlement (`com.apple.developer.background-tasks.continued-processing.gpu`). For simplicity
  and device parity across iPhone and iPad, we chose CPU-only background inference for MVP.
- **Expiration handler**: the system provides an `expirationHandler` closure when it needs the
  task to end; the app must stop work and call `setTaskCompleted(success:)` promptly.

### Why CPU Inference in Background?

Switching from GPU to CPU for background execution is necessary because:

1. Metal GPU contexts are not guaranteed to be available when the app is backgrounded.
2. Background GPU entitlement is iPad-only and not yet tested on device.
3. CPU inference, while slower, is fully functional and deterministic under BG QoS.
4. `LlamaCppService` supports hot reload via `unloadModel()` + `loadModel()`, making
   the GPU↔CPU switch safe as long as no inference is in-flight.

---

## 2. Decision

Implement opt-in background simulation continuation with the following constraints:

- **Default OFF**: the toggle is disabled by default each simulation session.
  Per-simulation reset avoids surprise background CPU usage.
- **iOS 26+ only**: gated via `if #available(iOS 26, *)`. Falls back to pause-on-background
  for iOS 17–25 (original ADR-002 §7 behaviour unchanged).
- **LlamaCppService only**: only `LlamaCppService` exposes `reloadModel(gpuAcceleration:)`.
  Other backends (Ollama, Mock) are always remote or in-memory — reload is meaningless.
- **CPU-only for background**: `n_gpu_layers: -1` (full GPU) in foreground;
  `n_gpu_layers: 0` (CPU only) in background. No background GPU entitlement required.
- **Foreground restore**: when the app returns to foreground, the model is reloaded back to
  GPU mode. Simulation pauses briefly during the reload, then resumes automatically.
- **Cooperative suspend/resume**: in-flight inference is interrupted via a
  `SuspendController` owned by the VM and attached to the LLM (see §4 / §10).
  This is required even for the Toggle OFF path to avoid Metal-denied decodes as
  the app transitions out of the active state.

## 3. Architecture Components

### 3.1 `GPUAcceleration` Enum (LLM/)

```swift
/// Controls Metal GPU layer offloading for LlamaCppService.
public enum GPUAcceleration: Sendable {
    case full   // n_gpu_layers = -1 (all layers on GPU)
    case none   // n_gpu_layers = 0 (CPU only)
}
```

Maps directly to `llama_model_params.n_gpu_layers`. Adding the enum keeps call sites
readable and avoids magic numbers at the reload site.

### 3.2 `LlamaCppService.reloadModel(gpuAcceleration:)` (LLM/)

```swift
func reloadModel(gpuAcceleration: GPUAcceleration) async throws
```

Sequential unload + reload with the new acceleration setting. Protected by
`generatingGuard` — a `precondition` that crashes if called during an in-flight
`generate()` call. Callers are responsible for pausing the simulation and confirming
the `.simulationPaused` event before invoking this method.

### 3.3 `BackgroundSimulationManager` (App/)

Wraps `BGTaskScheduler` and `BGContinuedProcessingTask`:

- `register()` — registers the task handler with `BGTaskScheduler` at app launch.
- `scheduleRequest(title:subtitle:onActivation:)` — submits a
  `BGContinuedProcessingTask` request. `onActivation` fires when the system activates
  the task after backgrounding; no expiration callback is exposed.
- `completeTask(success:)` — marks the active task complete and clears state. Safe
  to call when no task is active (no-op).
- `cancelPendingRequest()` — cancels a not-yet-activated request when the user
  disables the toggle.
- Internal `expirationHandler`: when the system expires the task, the manager
  synchronously calls `setTaskCompleted(success: false)` and clears state. The VM
  is not notified — see §5 and §10 for the rationale.

Placed in `App/` to satisfy the dependency rule: BackgroundTasks framework is an
App-layer concern; `LLM/` and `Engine/` must not import it.

### 3.4 `SimulationViewModel` (App/)

Entry points for suspend/resume orchestration:

| Method | Trigger | Action |
|--------|---------|--------|
| `enableBackgroundContinuation` | Toggle tapped ON in SimulationView | Calls `scheduleRequest(...)` on the manager |
| `handleWillResignActive()` | `UIApplication.willResignActiveNotification` | `suspendController?.requestSuspend()` — earliest signal (before `.background`) |
| `handleScenePhaseBackground()` | `scenePhase == .background` | Backstop `requestSuspend()` for direct `.active → .background` transitions |
| `handleBackgroundActivation()` | BG task activation callback (Toggle ON only) | `reloadModel(.none)` then `resume()` |
| `handleScenePhaseForeground()` | `scenePhase == .active` | Toggle ON (`isOnCPU`): re-`requestSuspend()`, `reloadModel(.full)`, `resume()`. Toggle OFF: just `resume()` |

The VM owns a fresh `SuspendController` per `run()` call and attaches it to the LLM
before `loadModel()`; see §10 for the full contract.

### 3.5 `SimulationView` Toggle (Views/)

A toggle button in the simulation controls bar, rendered only when
`#available(iOS 26, *)` AND the active `LLMService` is `LlamaCppService`.
Label: "Continue in Background". Default state: off.

The toggle is hidden entirely when unavailable to avoid confusing users on
older OS versions or when using Ollama/Mock backends.

## 4. Suspend/Resume Pattern (Critical)

### Why the original safe-reload pattern was insufficient

The original design relied on reaching a phase boundary (`SimulationRunner.checkPaused`)
before the OS denied GPU work, and used a `pauseAndAwaitConfirmation` continuation
to wait for the `.simulationPaused` event before reloading. Empirically, iOS denies
Metal decode within roughly 10-100 ms of `scenePhase = .background` — well below the
typical generate-to-phase-boundary transition time (several seconds per inference).
The in-flight `llama_decode` therefore failed with Metal error -3 before the pause
checkpoint ever fired, corrupting the KV cache and breaking the run.

`LlamaCppService.generatingGuard` still exists and still precondition-traps concurrent
`generate()` calls (ADR-002 §6) — the guard is orthogonal to the suspend mechanism.
What changed is how the VM achieves quiescence before reload: instead of waiting for a
phase boundary, it signals a cooperative `SuspendController` that the LLM layer
consults inside its auto-regressive loop.

### Suspend/resume sequence

```
UIApplication.willResignActiveNotification
        │                                       (scenePhase = .background is the backstop)
        ▼
SimulationViewModel.handleWillResignActive()
        │
        ▼
SuspendController.requestSuspend()              ← transitions state to .suspended
        │
        │  In-flight generate()'s auto-regressive loop checks
        │  isSuspendRequested() at each iteration; a live-suspend
        │  llama_decode failure is also converted reactively
        │  (llama_memory_clear wipes partial KV cache).
        ▼
throws LLMError.suspended
        │
        ▼
LLMCaller.generateWithSuspendRetry catches,
awaits SuspendController.awaitResume()
        │
        ├── Toggle ON ──────────────────────────────────────┐
        │                                                    │
        │   BG task activates (system side)                  │
        │           │                                        │
        │           ▼                                        │
        │   handleBackgroundActivation()                     │
        │           │                                        │
        │           ▼                                        │
        │   LlamaCppService.reloadModel(.none)               │
        │   (awaitGenerateIdle returns immediately — no      │
        │    in-flight generate because it's parked)         │
        │           │                                        │
        │           ▼                                        │
        │   SuspendController.resume()                       │
        │           │                                        │
        │           ▼                                        │
        │   LLMCaller retries the same prompt on CPU         │
        │           │                                        │
        │           ▼  (later: scenePhase = .active)         │
        │   handleScenePhaseForeground()                     │
        │   requestSuspend → reloadModel(.full) → resume     │
        │           │                                        │
        │           ▼                                        │
        │   LLMCaller retries on GPU                         │
        │                                                    │
        └── Toggle OFF ─────────────────────────────────────┐
                    │                                        │
                    ▼                                        │
           app is suspended by iOS                           │
                    │                                        │
                    ▼  (later: scenePhase = .active)         │
           handleScenePhaseForeground() → resume()           │
                    │                                        │
                    ▼                                        │
           LLMCaller retries on GPU. KV cache was wiped      │
           by the LLM-layer reactive path, so retry          │
           starts from clean state.                          │
                    │                                        │
                    ▼                                        │
           (normal run continues)                            │
                                                             │
        ─────────────────────────────────────────────────────┘
```

### Why suspend is cooperative + reactive

The generate loop can only check `isSuspendRequested()` between iterations. Between
iterations, a `llama_decode` that begins ~microseconds before the OS denies GPU work
will still run against a denied Metal context and fail with a non-zero error. Rather
than hard-coding Metal error -3 (fragile across llama.cpp and iOS versions), the
reactive path maps **any** non-zero `llama_decode` result under an active suspend
request to `LLMError.suspended`, after wiping the partial KV cache via
`llama_memory_clear`. This is the belt-and-braces that makes the mechanism robust
against future OS behaviour changes.

### `SimulationRunner.checkPaused()` call sites

Kept at round and phase boundaries — but **only for the user-initiated pause button**.
Scene-phase handlers no longer set `isPaused`. This preserves the pause button UX
while removing the original pattern's dependency on phase-boundary timing for
background safety.

## 5. State Machine: `SuspendController`

The `SuspendController` state machine governs the cooperative pause point inside the
LLM layer:

```
              requestSuspend()                resume()
      idle ────────────────────▶  suspended  ──────────▶  resumed
        ▲                             │                       │
        │       requestSuspend()      │                       │
        └─────────────────────────────┴───────────────────────┘
```

### Contract (enforced by `SuspendController` implementation)

- **Single awaiter only.** Concurrent callers to `awaitResume()` trap via
  `precondition`. The engine uses 1 generate = 1 waiter; anything else is a
  programming error.
- **Idempotent resume.** `resume()` is safe in any state, including when no suspend
  has been requested. This is essential for cleanup paths (`cancelSimulation()`,
  `run()`'s `defer`) that may race with a normal resume.
- **Cancellation honoured.** `withTaskCancellationHandler` releases the continuation
  when the owning `Task` is cancelled; callers check `Task.isCancelled` afterwards.
- **Re-armable.** After `resume()`, a subsequent `requestSuspend()` transitions back
  to `.suspended`, enabling multiple BG/FG cycles within a single run.

### Interaction with BG task expiration

BG task expiration is isolated from the `SuspendController` state machine by design:
the `BackgroundSimulationManager`'s internal `expirationHandler` synchronously calls
`setTaskCompleted(success: false)` and clears its own state. It does **not** notify
the VM and does **not** call `resume()`.

Why: late-firing expiration must not race with (or be mistaken for) a normal resume.
If expiration could trigger `resume()`, a task that expired after the user
foregrounded would spuriously re-fire and the suspend/resume cycle would lose
idempotence. Routing all `resume()` calls through VM-local lifecycle events
(`run()` defer, `cancelSimulation()`, `handleBackgroundActivation`,
`handleScenePhaseForeground`) makes the race structurally impossible.

The `SuspendController` therefore stays in `.suspended` across expiration; the user
foregrounding eventually drives `handleScenePhaseForeground → resume()`.

## 6. Performance Characteristics

### GPU↔CPU Reload Time

| Phase | Estimated Duration |
|-------|--------------------|
| `unloadModel()` | ~0.5-1s (free Metal buffers + weights) |
| `loadModel()` CPU | ~2-7s (re-read model file from disk, map weights to RAM) |
| **Total GPU→CPU** | **~3-8s** |
| **Total CPU→GPU** | **~3-8s** (same path, different n_gpu_layers) |

The 3-8s reload window is well within the `BGContinuedProcessingTask` expiration grace period
(which is not publicly documented but empirically observed at 30s+ for user-initiated tasks).

### CPU Inference Speed in Background

| Mode | Approx. Speed |
|------|--------------|
| Foreground GPU (A17/A18) | ~20-40 tok/s |
| Background CPU | ~2-8 tok/s (~0.1-0.2× GPU) |

The additional slowdown versus a typical CPU-foreground number is due to Background QoS
throttling: the OS assigns `.utility` or `.background` QoS to BG task threads, reducing
CPU scheduling priority.

**User expectation**: simulation continues, but noticeably slower. A progress indicator
showing "Running in background (slower)" is shown in the notification/Dynamic Island (if supported).

### Memory: GPU vs CPU

| Mode | Approx. RAM |
|------|------------|
| Foreground GPU (Q4_K_M, Metal buffers) | ~4.5-5.0 GB |
| Background CPU (Q4_K_M, no Metal) | ~3.5-4.5 GB |

CPU mode uses slightly less RAM (no Metal buffer allocation), reducing jetsam risk during
background execution. However, the app is still at risk if the OS needs the memory urgently.

### Thermal Throttling

The existing `throttleIfOverheating()` mechanism (ADR-002 §5) continues to apply in background.
At `.serious` or `.critical` thermal state, a 200ms pause is inserted between inferences.
CPU inference generates less heat than GPU, reducing the likelihood of sustained thermal
throttling during background execution.

## 7. Trade-offs Considered

### iPad GPU Background

**Status: Deferred**

iPad supports background GPU access via the
`com.apple.developer.background-tasks.continued-processing.gpu` entitlement (iOS 26+, iPad only).
This would allow ~20-40 tok/s background inference on iPad instead of ~2-8 tok/s CPU.

Deferred because:
- Requires an entitlement not yet requested or approved
- No iPad test device available for validation
- Complicates the implementation: GPU/CPU branch per device + per scene phase
- Device parity story is cleaner without it for MVP

Tracked as future work (§8).

### Automatic (Always-On) Background Continuation

**Rejected**

Making background continuation automatic (always enabled, no toggle) was rejected:
- Battery impact is significant — CPU inference at BG QoS still consumes meaningful power
- Background tasks require user initiation (OS enforces this)
- Users who do not want background AI activity should not be surprised by it

The toggle must be explicitly enabled before backgrounding.

### Per-App-Session vs Per-Simulation Toggle State

**Decision: Per-simulation (resets on each simulation start)**

Resetting the toggle for each simulation start is safer: users who forget they enabled it
for a previous run are not surprised. A persistent user preference (remember last state)
is deferred to Phase 2 (§8).

## 8. Future Work

- **iPad GPU background**: request `com.apple.developer.background-tasks.continued-processing.gpu`
  entitlement; add device-type check; test on physical iPad Pro M4+. Update ADR-003 when implemented.
- **Persistent toggle preference**: remember last background continuation state across app launches
  via `UserDefaults`. Reset on first launch of a new app version.
- **Thermal-aware dynamic throttling**: in background, monitor thermal state and adjust inter-inference
  delay dynamically (e.g., 500ms at `.serious`, yield task at `.critical`).
- **Checkpoint save on backgrounding**: serialize `SimulationState` to disk before entering background
  so a jetsam kill during suspension does not lose round progress. Assess complexity before committing.
- **Dynamic Island / Live Activity**: show simulation progress (current round / total) while backgrounded.
  Requires ActivityKit entitlement (separate PR).

## 10. Suspend/Resume Contract

Authoritative summary of the contract each layer upholds. Changes here require a
matching ADR update.

### VM (App layer)

Creates a fresh `SuspendController` per `run()` call and attaches it to the LLM before
`loadModel()`. Stores the controller until `run()` exits; clears it in the `defer`
block alongside other per-run state.

Calls `requestSuspend()` in:

- `handleWillResignActive()` — earliest signal, fires before `scenePhase = .background`
- `handleScenePhaseBackground()` — backstop for `.active → .background` transitions
- `handleScenePhaseForeground()` just before `reloadModel(.full)` (Toggle ON return)

Calls `resume()` in:

- `run()`'s `defer` block — covers normal exit, LLM-load failure, and cancellation
- `cancelSimulation()` — belt-and-suspenders wake for a parked awaiter
- `handleBackgroundActivation()` — after `reloadModel(.none)`
- `handleScenePhaseForeground()` — after `reloadModel(.full)` on Toggle ON, or
  directly on Toggle OFF

### LLM layer (`LlamaCppService`)

- `attachSuspendController(_:)` stores the controller reference. Preserved across
  `reloadModel` via `defer` so the VM can keep the same reference after a GPU↔CPU
  swap (see PR #84 Step 11).
- Cooperative check: the auto-regressive generate and prefill loops poll
  `isSuspendRequested()` at iteration boundaries; `true` → throw
  `LLMError.suspended`.
- Reactive path (`decodeFailureError(_:)`): any non-zero `llama_decode` result
  coinciding with `isSuspendRequested() == true` wipes the partial KV cache via
  `llama_memory_clear` and returns `.suspended`. Mapping is intentionally
  code-agnostic (no hard-coded Metal error number) for robustness across llama.cpp
  and iOS versions.

### Engine layer (`LLMCaller`)

- `call(...)` takes `suspendController` as a required parameter.
- `generateWithSuspendRetry` catches `LLMError.suspended` → `awaitResume()` →
  `Task.checkCancellation()` → retry the same prompt.
- Suspend cycles are invisible to the parse-retry counter (retry budget is not
  consumed) and to `inferenceStarted` / `inferenceCompleted` emissions (the UI
  shows a single "thinking" indicator across a suspend/resume cycle).

### BG task manager (`BackgroundSimulationManager`)

- `scheduleRequest(title:subtitle:onActivation:)` — activation-only; no expiration
  callback is exposed.
- Internal `expirationHandler`: synchronously calls `setTaskCompleted(success: false)`
  and clears state. Does **not** signal the VM.
- `completeTask(success:)` — called by the VM on foreground return; idempotent.

### Global invariants

1. Exactly one `SuspendController` per `run()` call — never shared across runs.
2. `resume()` is idempotent across all states (`.idle` / `.suspended` / `.resumed`).
3. At most one awaiter on `awaitResume()` — concurrent awaiters `precondition`-trap.
4. BG task expiration never calls `resume()` — only VM-local lifecycle does.
5. `llama_decode` → `.suspended` mapping is driven by the live suspend flag, not
   by any specific error code.
6. `runner.isPaused` and `SuspendController.requestSuspend()` are co-managed
   by the unified `pauseSimulation(reason:)` / `resumeSimulation()` API on
   `SimulationViewModel`. Views MUST NOT mutate `isPaused` directly — the
   property is read-only externally. The pause button and the memory-warning
   handler both call this API so an in-flight `generate` is interrupted
   cooperatively rather than waiting for the next phase boundary. Scene-phase
   handlers continue to call `requestSuspend()` only (no `isPaused` change),
   since they don't expose a user-visible paused state.
