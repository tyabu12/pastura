# ADR-002: llama.cpp as Interim On-Device LLM Backend

> **Status:** Accepted
> **Date:** 2026-04-08
> **Context:** Phase 1 MVP requires on-device LLM inference for TestFlight distribution.
> LiteRT-LM Swift SDK is not yet released; its C API lacks iOS GPU support.

## Summary

Use llama.cpp (via Metal GPU) as the interim production LLM backend for TestFlight.
Migrate to LiteRT-LM when the official Swift SDK ships with iOS GPU support.

---

## 1. Why Not LiteRT-LM Today?

### LiteRT-LM Status (as of 2026-04-08)

| SDK | Status |
|-----|--------|
| Python | Stable |
| Kotlin (Android) | Stable |
| C++ | Stable |
| **Swift** | **In Dev ("Coming Soon")** |

- Swift SDK: no SwiftPM package, no documentation, no release timeline.
- C API (`c/engine.h`): works on iOS but **GPU inference is unavailable** on iOS.
  iOS prebuilts are missing entirely ([LiteRT-LM Issue #1050](https://github.com/google-ai-edge/LiteRT-LM/issues/1050)).
  CPU-only mode yields ~9.7 tok/s for E4B (the larger 4B variant) on iPhone — usable but suboptimal.
- AI Edge Gallery iOS app exists on App Store (uses LiteRT-LM C/C++ API internally)
  but its source code is not public ([gallery Issue #420](https://github.com/google-ai-edge/gallery/issues/420)).

Reference: [Zenn article on LiteRT-LM iOS integration](https://zenn.dev/yoshitetsu/articles/ac05dc4a71650d)
documents the C API build process (Bazel, 609 static libs, 360 MB unified archive, `-force_load`,
16 MB stack allocation, dylib-to-framework conversion).

### Decision

LiteRT-LM's C API path is viable but high-effort with no GPU payoff on iOS today.
llama.cpp offers Metal GPU support, mature iOS tooling, and Gemma 4 day-one support.
Use llama.cpp as the interim backend.

## 2. SwiftPM Package Selection

### Evaluation Criteria

- iOS platform support (`.iOS(.v16)` or later)
- Metal GPU shader compilation for iOS
- Swift 6.x strict concurrency compatibility
- Upstream llama.cpp tracking (Gemma 4 requires recent builds)
- Active maintenance

### Candidates Evaluated

| Package | iOS | Swift | Upstream | Last Updated | Verdict |
|---------|-----|-------|----------|-------------|---------|
| **mattt/llama.swift** | `.iOS(.v16)` | 6.0 | b8694 (daily) | 2026-04-07 | **Selected** |
| Official xcframework | N/A (manual) | N/A | Current | Daily releases | Fallback |
| tattn/LocalLLMClient | `.iOS(.v17)` | 6.1 | b6871 (~1800 behind) | 2026-01-31 | Too stale |
| eastriverlee/LLM.swift | `.iOS(.v16)` | 5.9 | Unknown | 2025-12-06 | Too stale |
| StanfordBDHG/llama.cpp | `.iOS(.v17)` | 5.9 | 2+ years stale | 2024-05-12 | Dead |

### Decision

**Primary: [mattt/llama.swift](https://github.com/mattt/llama.swift)**

- Thin SPM wrapper that re-exports the official llama.cpp xcframework from each release
- Tracks upstream within hours — Gemma 4 support guaranteed
- Zero extra dependencies — just the C API, which is exactly what `LlamaCppService` needs
- Created by Mattt (NSHipster), established Swift community member

**Fallback: Official xcframework direct consumption**

If mattt/llama.swift becomes unmaintained, download the xcframework directly from
[llama.cpp releases](https://github.com/ggml-org/llama.cpp/releases) and add as a
local binary SPM target. This is what mattt/llama.swift automates.

### Pin Strategy

Pin to a **specific release tag** (not `main` branch) to ensure reproducible builds.
PR1 should select the latest tag at implementation time and verify Gemma 4 E2B stability:
model loads without crash, JSON output parses successfully, no hangs over 10+ inferences.
Document the chosen tag in the PR description.

## 3. Model Specification

### Gemma 4 E2B

| Property | Value |
|----------|-------|
| Effective parameters | 2B (active during inference) |
| Total parameters | 5.1B (Per-Layer Embeddings) |
| Quantization | Q4_K_M |
| Download size | ~3.1 GB |
| Runtime RAM | ~5 GB (weights + KV cache + Metal buffers) |
| Context limit | 8192 tokens (configured, not model max) |
| GGUF source | [unsloth/gemma-4-E2B-it-GGUF](https://huggingface.co/unsloth/gemma-4-E2B-it-GGUF) (Q4_K_M) |
| License | Apache 2.0 |

### Why Q4_K_M?

- ~4.5 bits/param average — best quality/size balance for on-device
- ~3.1 GB download is practical for mobile (< App Store 4 GB OTA limit)
- ~5 GB RAM fits 8 GB devices, but headroom is tight (~0.5-1.5 GB after iOS)
- Memory warnings should be expected; `didReceiveMemoryWarning` handling is critical
- Q2_K (~1.5 GB) has significant quality loss; Q8_0 (~5.4 GB) wastes bandwidth/RAM

### Why 8192 Context Limit?

- Pastura simulations rarely exceed 8K tokens of accumulated context
- Bounds KV cache memory (~200 MB at Q8 KV for 8K vs ~800 MB for 32K)
- Bounds prefill time (~20-40s for 8K vs potentially minutes for 32K)
- Can be relaxed later based on real-device profiling

### Model Fallback Chain

If Gemma 4 E2B GGUF proves unstable on llama.cpp (support is <1 week old),
fall back to an older, well-tested model:

1. **Gemma 3 4B Q4_K_M** (~2.5 GB, well-established llama.cpp support)
2. **Gemma 2 2B Q4_K_M** (~1.5 GB, oldest and most stable)

## 5. Performance Expectations

### Inference Speed (estimated, llama.cpp + Metal GPU)

| Metric | iPhone 15 Pro (A17) | iPhone 16 Pro (A18) |
|--------|--------------------|--------------------|
| Decode | ~20-35 tok/s | ~25-40 tok/s |
| Prefill (4K tokens) | ~200-400 tok/s | ~250-500 tok/s |

### Comparison with LiteRT-LM (reference)

| Backend | Decode (E2B, GPU) | Notes |
|---------|-------------------|-------|
| LiteRT-LM | ~56.5 tok/s | iPhone 17 Pro benchmark (newer HW) |
| **llama.cpp** | **~20-40 tok/s** | **iPhone 15/16 Pro est.; gap may narrow on same HW** |

### Simulation Time Estimates

For a typical 50-inference simulation (~200 tokens output each, ~2K avg prompt):

| Phase | Time |
|-------|------|
| Prefill (cumulative) | ~3-5 min |
| Decode (cumulative) | ~4-8 min |
| Thermal throttling (20-40% slowdown after 2-5 min) | +2-4 min |
| **Total estimate** | **~10-20 min** |

ADR-001 §8 quotes "50 inferences ≈ 2.3 min" based on LiteRT-LM decode-only math.
Real-world llama.cpp performance is 4-8x slower due to: (a) lower decode speed,
(b) prefill time not previously accounted for, (c) thermal throttling on iPhone.

### Thermal Throttling Mitigation

- Monitor `ProcessInfo.processInfo.thermalState`
- At `.serious` or `.critical`: insert 200ms pause between inferences
- Log thermal state transitions for TestFlight diagnostics

## 6. Architecture: LlamaCppService

### Layer Placement

```
LLM/
├── LLMService.swift          (protocol — unchanged)
├── OllamaService.swift       (dev — unchanged)
├── MockLLMService.swift      (test — unchanged)
├── LlamaCppService.swift     (NEW — llama.cpp backend)
├── JSONResponseParser.swift  (unchanged)
└── LLMError.swift            (unchanged)

App/
├── ModelManager.swift         (NEW — model download/management)
├── AppDependencies.swift      (UPDATED — deferred LLMService init)
└── ...
```

### Why ModelManager in App/, not LLM/?

`LLM/` depends on `Models` only (CLAUDE.md dependency rules). `ModelManager` handles:
- HTTP download (URLSession)
- File system management (Documents directory)
- Device capability checking (ProcessInfo)
- Download progress reporting (for UI)

These are App-layer concerns. `LlamaCppService` receives a model file path
via its constructor (`init(modelPath: String)`), mirroring how `OllamaService`
receives `baseURL`.

### Concurrency Contract

Matching OllamaService (ADR-001 §7):

- Engine executes inferences sequentially — no concurrent `generate()` calls
  (a `precondition` catches any regression).
- `loadModel()` / `unloadModel()` cooperatively wait (polling `generatingGuard`)
  if a `generate()` is in flight. Historically this was a `precondition` crash,
  but llama.cpp's C API does not respect Swift `Task` cancellation — `generate()`
  always runs to completion even if its owning task is cancelled, so the crash
  fired on legitimate cleanup paths (memory warning, user cancel mid-generate).
- Thread safety for `isModelLoaded` state: `OSAllocatedUnfairLock<Bool>`.

### Chat Template and Output Format

**Chat template**: Use llama.cpp's built-in chat template application for Gemma 4.
llama.cpp reads the template from the GGUF metadata and applies it automatically
via `llama_chat_apply_template()`. `LlamaCppService.generate(system:user:)` passes
the system and user strings; llama.cpp formats them into Gemma's
`<start_of_turn>user\n...<end_of_turn>` format internally.

**JSON output**: Continue using the existing retry-based `JSONResponseParser` approach
(same as OllamaService). Do NOT use GBNF grammar constraints for MVP — the parser
already handles Gemma's thinking tags and noisy output reliably. GBNF grammar may be
explored in Phase 2 if JSON parse failure rates are high in TestFlight telemetry.

### Sampling Parameters

Match OllamaService defaults:
- Temperature: 0.8
- Max tokens: 1000
- Top-k and top-p: use llama.cpp defaults (top-k=40, top-p=0.95)
- Repeat penalty: use llama.cpp default (1.1)

### AppDependencies Deferred Initialization

Current `AppDependencies` creates `llmService` eagerly in `init()` as a `let` property.
For `LlamaCppService`, creation must wait until the model file is on disk.

Solution: use the existing `PasturaApp.dependencies: AppDependencies?` pattern.
- `nil`: show model download screen
- After download: create `AppDependencies` with `LlamaCppService(modelPath:)`
- `let llmService` stays immutable — no change to Sendable reasoning

## 7. App Lifecycle

### Background/Foreground Transitions

When iOS suspends the app, Metal GPU buffers may be invalidated by the OS.
Resuming with stale buffers causes `EXC_BAD_ACCESS` or corrupt output.

| Event | Action |
|-------|--------|
| `scenePhase == .background` | Pause simulation (existing `isPaused` flag) |
| `willResignActive` | Do NOT unload model (would lose KV cache state) |
| Return to foreground | Validate Metal context; if invalid, unload + reload model |
| `didReceiveMemoryWarning` | Cancel simulation, then `unloadModel()` (waits for in-flight generate) |

### Checkpoint Safety

Consider saving simulation checkpoint state before entering background,
so a jetsam kill during suspension doesn't lose progress. (May defer to Phase 2
if complexity is high.)

## 10. Streaming Extension (2026-04-17 amendment)

Context: the Phase 2 "Real-time LLM token streaming" feature (Issue #119)
needed to replace pseudo-typing with actual token-by-token output from
llama.cpp while keeping the llama.cpp → LiteRT-LM migration cheap.
Critic pre-mortem review informed the partial-JSON best-effort strategy.

### 10.1 Protocol additions

`LLMService` now declares:

```swift
func generateStream(system: String, user: String)
  -> AsyncThrowingStream<LLMStreamChunk, Error>
```

and

```swift
public struct LLMStreamChunk: Sendable, Equatable {
  public let delta: String
  public let isFinal: Bool
  public let completionTokens: Int?
}
```

A default implementation in a protocol extension wraps
`generateWithMetrics` and yields a single terminal chunk — so
`MockLLMService` (with no streaming chunks configured) and
`OllamaService` satisfy the contract without code changes.

Contract invariants:
- Exactly one chunk per stream has `isFinal == true`, and it is always
  the last chunk observed.
- `completionTokens` is populated only on the final chunk, and only
  when the backend can cheaply report it (llama.cpp: yes; MediaPipe /
  LiteRT-LM: final-chunk only per shipping API).
- Deltas never contain the stop sequence `<|im_end|>` — llama.cpp's
  stream strips it in-place; callers concatenate deltas verbatim.

### 10.2 Field set is deliberately narrow

No backend-specific fields (internal token IDs, sampler stats, decode
timing) appear on `LLMStreamChunk`. LiteRT-LM's shipping analog
(`generateResponseAsync`) exposes only `delta: String`; the optional
`completionTokens: Int?` is already set up to gracefully degrade to
`nil` on backends that don't report per-inference token counts. This
means the §8 migration claim ("replace `LlamaCppService.swift` with
`LiteRTLMService.swift`, no changes to Engine, Views, or App layer")
extends to the streaming path: a `LiteRTLMService.generateStream`
implementation either calls LiteRT-LM's async sequence directly or
wraps its callback API via `AsyncThrowingStream { continuation in ... }`
with no protocol change.

### 10.3 Partial-JSON parse strategy

`PartialOutputExtractor` (`Pastura/LLM/PartialOutputExtractor.swift`)
runs on every accumulated buffer and returns
`(primary: String?, thought: String?)` snapshots. Key rules:

- Does not emit primary content until past the opening quote of a
  recognised primary key (`statement`, `declaration`, `boke`, `action`,
  `vote`) — shrinks responsibility to the tractable "inside a known
  string value" case and eliminates the "leading garbage rendered as
  primary" failure mode the critic flagged.
- Strips closed Gemma channel-thinking and generic `<think>` tags;
  while an *unclosed* tag is present, yields an empty snapshot (the
  JSON may be inside the model's reasoning).
- Top-level key detection uses a dual check (preceded by `{`/`,` AND
  followed by `:`) so key-like substrings inside string values do not
  falsely trigger (e.g., `{"inner_thought":"statement"}`).
- Handles JSON escapes (`\"`, `\\`, `\uXXXX`, etc.) and holds back
  incomplete trailing escapes.
- `LlamaCppService` holds back up to three trailing bytes before
  yielding decoded text so a multi-byte UTF-8 character split across
  two token pieces never surfaces as a replacement character.

Final canonical parse still runs on stream end via `JSONResponseParser`
— the partial extractor never replaces it. On divergence,
`SimulationViewModel` logs a debug-level telemetry event.

### 10.4 Rejected alternative: dual-channel

"Model emits plain primary text, then separator, then JSON metadata"
was rejected. Phase 0 spent significant effort on JSON-output
stability across all three presets × all eight phase types; a format
change would require Phase 0-scale revalidation (and would also touch
in-flight user-authored scenarios via the Share Board). The
partial-JSON path is additive, rolls back behind a feature flag
(`FeatureFlags.realtimeStreamingEnabled`, UserDefaults), and requires
zero prompt changes.

### 10.5 Event contract

New `SimulationEvent.agentOutputStream(agent, primary, thought)` fires
per partial snapshot. Consumers overwrite their per-agent buffer on
each emission — a new stream (retry after parse failure, re-issue
after suspend/resume) naturally replaces prior snapshots without a
separate reset event. `.agentOutput` still fires exactly once at
stream end with the final parsed `TurnOutput`; handlers and
persistence consume from that event as before.

### 10.6 Rollout and safety hatch

Feature flag `realtimeStreamingEnabled` (default `true`) gates the
SimulationViewModel consumer. If the partial-parser path surfaces a
pathology in TestFlight (divergence rate > 1%, visible regressions in
preset runs), the flag flips off and the UI falls back to the
pre-streaming flow — no release required.

Related code:
- `Pastura/LLM/LLMStreamChunk.swift`
- `Pastura/LLM/PartialOutputExtractor.swift`
- `Pastura/LLM/LlamaCppService.swift` (streaming extension)
- `Pastura/LLM/MockLLMService.swift` (`setStreamChunks`)
- `Pastura/Engine/LLMCaller.swift` (stream consumer + retry)
- `Pastura/App/FeatureFlags.swift`
- `Pastura/App/SimulationViewModel.swift` (snapshot state, telemetry)
- `Pastura/Models/SimulationEvent.swift` (new case)

## 11. Streaming Display Redesign Outcome (2026-04-21 amendment)

### 11.1 Context

The §10 streaming rollout (#119 → PR #132) surfaced two on-device UX
bugs in TestFlight QA:

- Line-break jitter mid-stream as `Text` re-laid out on every token.
- Same-speaker streaming row occasionally vanishing and re-appearing
  with different content ("text vanishes mid-stream").

Issue #133 tracked the redesign with three critic-informed axes
(view architecture, jitter mitigations, `.instant` semantics) and a
pivot matrix (`A` parse-retry silent / `B` LazyVStack `@State` recycle /
neither). PR #158 (PR#4) landed DEBUG-only diagnostic instrumentation
+ an analyzer script (`scripts/analyze-streaming-diag.sh`) to measure
the three hypotheses on-device. During PR#4 validation a fourth
hypothesis, `A′` silent stream re-issue via `.suspended`, surfaced
(§11.4). This amendment records the measurement outcome, the pivot
decision it supports, and the UX treatment it requires.

### 11.2 Axis decisions (with outcomes)

#### Axis ① — View architecture

**Decision**: C′ (single-source-of-truth via trailing-slot LogEntry
with a separate `StreamingLogEntryRow`) — **deferred, not retired**.

Rationale: Hyp B was the load-bearing premise for C′. Four PR#4
sessions produced zero recycle events (§11.3). `.id()` stability
across turns was observed (session 1) but rendered benign by the
concat-trick clamp `min(visibleChars, primaryLen)`, so the `@State`
persistence does not misdisplay content.

**Re-open trigger**: Hyp B's documented reproduction ("scroll a
committed row off-screen and back while streaming continues" — Issue
#133 §10 Hypothesis B) was **not** explicitly exercised in the PR#4
sessions. Re-open Axis ① if any of the following surface:

- TestFlight telemetry captures an `AgentOutputRow` `onAppear` with
  `visibleChars` that previously advanced resetting to 0 (the DEBUG
  `StreamingDiag` logger is removable without losing this signal —
  a lightweight production counter could be added if needed).
- User report matching "committed row re-types itself or blanks out
  during a long simulation with scroll-up."
- Any future PR introducing heavier LazyVStack content (e.g., event
  injection, reflection phase with nested rows) where the state
  footprint per row grows.

The state-hoist + trailing-slot fix surface remains tractable; keeping
it unimplemented until Hyp B reproduces trades hypothetical cleanup
for concrete work the measurement does not currently justify.

#### Axis ② — Jitter mitigations

**Decision**: B1 + B3 landed (PR #154), B5 cancel-race landed (PR #147
+ #150) and verified clean under exercise (§11.3 session 2/3 B5
bucket (d) = 0). B2 word-boundary buffering remains **rejected**
(Japanese-first corpus has no whitespace signal). B4 TextKit 2 remains
**deferred** (Dynamic Type / accessibility mapping cost).

B6 (sentence-boundary buffering, raised in Issue #133 comment
`4275039657` as a CJK-appropriate alternative to B2) stays on the
shelf — no jitter-mitigation gap currently observed that would justify
the LLM-layer emit-policy change. If B1+B3 prove insufficient on a
lower-tier device class (iPhone SE-class pushing reveal cps above the
device's frame budget), B6 is the next natural step before B4.

#### Axis ③ — `.instant` enforcement

**Decision**: Choice 2 (event-layer gate in
`SimulationViewModel.handleAgentOutputStream`) remains the chosen
layer. Implementation is **pending** as the master tracker's PR#6.

Pivot-robustness note: the event-layer gate is independent of the
Axis ① outcome. It is equally valid under Option 0 (selected) and C′
(deferred), because the decision concerns *where* `.instant` short-
circuits, not *how* the streaming row is structured.

Accepted trade-off (unchanged from master tracker): `.instant` users
lose the live **snapshot-vs-canonical divergence check** in
`handleAgentOutput` — that branch requires a non-nil
`streamingSnapshot`, which never holds under `.instant`. The upstream
DEBUG `detectSilentStreamReIssue` diagnostic (Hyp A′ signal; runs at
the `handleEvent` dispatch site, above the `.instant` gate) is
distinct from this check and unaffected by the gate — it continues
to fire for all speeds. Alternative coverage for the lost
snapshot-vs-canonical check: (a) `.normal`/`.fast` user telemetry
(majority), (b) Ollama integration tests enforcing partial-prefix
invariant, (c) `PartialOutputExtractor` unit tests.

### 11.4 Hyp A′ finding — silent stream re-issue (discovered during PR#4)

Hyp A′ was not in the Issue #133 body's original (A / B / neither)
enumeration. It surfaced during PR#4 validation on 2026-04-20 when
user-observed same-agent content resets outnumbered the retry-counter
firings (distinct from the session-3 ミサキ parse retry tabulated in
§11.3, which the retry counter *did* catch — this is about earlier
observations on sessions 1/2 where the retry counter reported 0
despite visible resets). Root-cause
tracing (see `scripts/analyze-streaming-diag.sh` lines 79-106, 224-226,
and `SimulationViewModel.detectSilentStreamReIssue`) pointed at
`LLMCaller.consumeStreamWithSuspendRetry`:

```swift
// Pastura/Engine/LLMCaller.swift:152–183 (abridged)
while true {
  var rawText = ""
  let stream = llm.generateStream(system: system, user: user)
  do {
    for try await chunk in stream { /* emit .agentOutputStream */ }
    return StreamResult(rawText: rawText, completionTokens: ...)
  } catch LLMError.suspended {
    await controller.awaitResume()
    try Task.checkCancellation()
    // Loop: re-issue a fresh stream. rawText resets on the next
    // iteration, so the extractor starts from "" and the visible
    // primary resets too — but `.inferenceStarted` is NOT re-fired.
  }
}
```

The outer retry loop in `LLMCaller.call` emits `.inferenceStarted`
per `for attempt in 0...maxRetries` iteration, catching JSON parse
retries (Hyp A). The *inner* re-issue after `.suspended`, however,
bypasses that emission — a silent stream restart. Hyp A′ detection
exists only because PR#4's `lastRawStreamingPrimary` tracker compares
successive `.agentOutputStream` primaries and flags the shrink or
divergence that results from the re-issued stream's fresh `rawText`.

Session 4 confirmed Hyp A′ reproduces *every* time Control Center is
pulled during an active stream: 4 events in a single ケンタ turn
across ~40 seconds, demonstrating that the silent re-issue is a
deterministic consequence of scene-phase transitions — not a rare
edge case.

The detector (`detectSilentStreamReIssue`) is DEBUG-only and operates
on raw pre-ContentFilter primary to avoid false positives from filter
rewrites (`"fuck" → "***"` would otherwise register as a diverge).
It remains in place post-(β) so continued telemetry tracks the
effectiveness of any source-layer fix landed in #135.

### 11.5 Pivot decision — 2×2 outcome

Rather than the master tracker's original 3-way pivot (A / B /
neither), the measurement plus the Hyp A′ discovery call for a 2×2
decomposition — one row per independently-firing symptom, one column
per axis of response:

| Fired? | Detection | UX treatment | Source-layer owner |
|---|---|---|---|
| **Hyp A** (1 session, `ミサキ`) | `retry agent=X attempt=N` counter in VM | (β1) pin-latest-and-replace — §11.6 | JSONResponseParser robustness + prompt stability (no current work) |
| **Hyp A′** (1 session, `ケンタ` ×4) | `streamReset agent=X type=shrink\|diverge` in VM | same (β1) — identical visual symptom, unified treatment | `LLMCaller.consumeStreamWithSuspendRetry` (#135) |
| **Hyp B** (0 sessions) | `debugInstanceID` UUID diff on row `onAppear` | — (deferred) | state hoist + trailing slot (deferred, PR#7/8a/8b superseded) |

Both A and A′ fire in the measurement window; the same UX treatment
addresses both because the user-visible symptom is identical
("streaming row resets mid-content"). The detection channels stay
distinct because they signal different root causes (parse robustness
vs BG suspend-resume) with different long-term fix owners.

The current `realtimeStreamingEnabled = false` rollback hatch remains
the safety net if (β1) itself regresses in TestFlight.

### 11.6 UX treatment — (β1) pin-latest-and-replace

Three treatments were on the table per Issue #133 Section
"Proposed approaches" A:

- **(α)** Reuse thinking indicator with a "retrying" / "resuming"
  variant.
- **(β)** Keep failed partial pinned (desaturated) and replace
  in-place on the first new partial.
- **(γ)** Emit a log entry for the failed attempt with a "retry"
  annotation; render the retry as a fresh row.

**Selected**: **(β1) — pin the *latest* partial before each reset,
replace on the first new non-empty partial.**

Semantics under repeated re-issues (session 4 had 4 in one turn):

- On `streamReset` (detected via the A′ tracker, or on `.inferenceStarted`
  for A): the existing `streamingSnapshot` is moved to a per-agent
  `pinnedPartial` slot and rendered at ~50 % saturation. Agent-turn-
  keyed, so each new `.inferenceStarted` for the same agent clears
  the pin.
- First new non-empty `.agentOutputStream` primary for the agent:
  the pin is cleared, a fresh streaming row renders the new partial
  from 0, and reveal proceeds as normal.
- Repeated reset during one turn (N resets): each reset overwrites
  `pinnedPartial` with the *most recent* pre-reset partial. The user
  sees the desaturated pin swap per reset; the streaming row that
  replaces it each time shows the latest re-issued content. Simpler
  than (β2) "pin-longest" which would require carrying the per-agent
  max-length partial across the turn; the (β1) semantics are always
  "what was visible just before the latest interruption."

Rejected:

- **(α)** was considered but rejected because the thinking-indicator
  variant loses the partial content already shown. For a 160-char
  partial already on screen (session 4 event 1), swapping to
  "resuming…" discards visible progress and subjectively feels like
  a bigger reset than the current "streaming row vanishes" UX.
- **(β2)** pin-longest was considered; rejected because it requires
  per-agent-turn state beyond the existing snapshot, and because in
  the divergent-reissue case the "longest prior partial" is content
  that will never be finished and may visually conflict with the
  new content that eventually commits.
- **(γ)** was rejected because a fresh row per failed attempt
  violates the append-only invariant for *this turn's* committed
  row; the committed LogEntry should still represent the one
  successful commit, not each attempt.

**Retirement condition for (β1)**: if #135 ships a partial-preserving
resume (or emits a first-class `streamReIssued(agent:)` event that
the VM can consume to distinguish "stream continues" from "stream
restarted"), the Hyp A′ branch of (β1) becomes dead code and the
display-layer state can be simplified. The detector
(`detectSilentStreamReIssue`) already rides on raw primaries that
survive any such Engine-layer change; it provides the continued
telemetry needed to measure #135's effectiveness.

Parse-retry (Hyp A) is orthogonal to #135 and remains (β1)'s
responsibility indefinitely.
