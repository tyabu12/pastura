## 1. プロダクト概要

### 1.1 コンセプト

Pastura（ラテン語で「放牧」）は、スマホ上でAIマルチエージェントシミュレーションを手軽に実験できるプラットフォーム。

学術研究（Generative Agents, Project Sid）で行われているマルチエージェントシミュレーションを、ローカルLLMのコストゼロ特性を活かして誰でもスマホで無料・無限に試せる。

ユーザーはシナリオ（ルール・ペルソナ・お題）を自由に設計し、AIエージェントを放牧して何が起きるか観察する。面白さはLLMの出力品質ではなく、ユーザーの実験設計と予想外の結果から生まれる。

### 1.2 ポジショニング

| 既存 | Pastura |
|------|---------|
| 研究者がクラウドAPIで高コストで実行 | 誰でもスマホで無料・無限に実行 |
| 1回の実験にAPIコスト数千円〜 | 完全オフライン、コストゼロ |
| コード実装が必要 | 宣言的なシナリオ定義だけで実行可能 |
| 論文のための実験 | 好奇心のための実験 |

### 1.3 コアメタファー

**放牧（Pastura）**。ユーザーは牧場主。AIエージェントという動物を牧場（シナリオ）に放ち、勝手に動き回る様子を観察する。llama, alpaca, vicuna… LLM界の動物名とも通じる世界観。

### 1.4 ピボットの経緯

E2B実機テスト（Ollama + Gemma 4 E2B on M1 MacBook）で以下を確認:
- 囚人のジレンマ: 構造化された判断は可能だが、戦略の深みや学習が不足。「微妙」
- ボケて: ユーモア生成はE2Bクラスでは品質不足。「説明」になり「ボケ」にならない
- JSON出力の安定化は達成（プロンプト改善 + リトライ機構で無言問題を解消）

結論: E2Bの出力品質に「面白さ」を依存するモデルはリスクが高い。代わりに「実験の手軽さ」を価値の中心に据える。

## 2. MVPで検証する仮説

### 2.1 コア仮説

> **「自分でシナリオを設計してAIマルチエージェントシミュレーションを回せるとしたら、ユーザーは試してみたいと思うか？」**

### 2.2 サブ仮説

- 宣言的シナリオ定義（YAML）で、技術者層は直感的にシナリオを作れるか？
- 外部LLM（Claude等）でシナリオ定義を自然言語から生成するワークフローは実用的か？
- E2Bの出力品質が低くても、「実験結果」として観察する分には価値があるか？
- シナリオをSNSで共有したくなる動機は生まれるか？

### 2.3 Go/No-Go 基準（Phase 1完了時）

**Go（Phase 2に進む）条件:**
- テスター5人中3人以上が「自分でもシナリオを作ってみたい」と回答
- プリセット以外のユーザー作成シナリオが1つ以上テスター自身から生まれる
- 「これ○○でやったら面白そう」という発話がテスト中に自然発生

**No-Go（撤退 or 再ピボット）条件:**
- テスターの反応が「ふーん、で？」止まりで実験への興味が湧かない
- シナリオ定義のハードルが高すぎてテスターが自作を諦める
- E2Bの出力がノイズすぎて実験結果として観察する意味がない

## 4. シナリオエンジン設計

### 4.1 設計思想

**「Terraform for AI Experiments」** — シナリオをコードではなく宣言的な設定ファイルで定義する。

設計原則:
- LLMにはテキスト生成だけをやらせる
- ルール進行・スコア計算はコードが確実に処理する
- シナリオ追加は設定ファイルのみ（コード変更不要）
- 外部LLMで自然言語からシナリオ定義を生成できる

### 4.2 フェーズタイプ一覧

| タイプ | 説明 | 処理主体 |
|--------|------|----------|
| `speak_all` | 全員が1つの発言を生成 | LLM |
| `speak_each` | 指名された1人が順番に発言 | LLM |
| `vote` | 全員が誰か1人に投票 | LLM |
| `choose` | 選択肢から1つ選ぶ | LLM |
| `score_calc` | スコアを計算 | コード |
| `assign` | エージェントに情報を配布 | コード |
| `eliminate` | 最多得票者を脱落 | コード |
| `summarize` | 結果を要約テキスト化 | コード |

**Phase 2以降:**
- `conditional` — 条件分岐
- `event_inject` — ランダムイベント注入
- `reflect` — エージェントの自己振り返り

### 4.3 シナリオ定義フォーマット

```yaml
id: prisoners_dilemma
name: 囚人のジレンマ
description: 協力か裏切りか。宣言と行動の裏表を楽しむ戦略ゲーム
agents: 5
rounds: 10

context: |
  あなたはゲーム番組「囚人のジレンマ」の出場者です。
  対戦相手と「協力」か「裏切り」を選びます。
  得点: 協力×協力=3,3 / 協力×裏切り=0,5 / 裏切り×裏切り=1,1

personas:
  - name: アキラ
    description: 冷静な戦略家。データから最適解を計算する
  - name: ミサキ
    description: お人好しの楽観主義者。基本的に人を信じる

phases:
  - type: speak_all
    prompt: "全員に向けて1文だけ宣言してください。ブラフも可。"
    output: { declaration: string, inner_thought: string }

  - type: choose
    pairing: round_robin
    prompt: "対戦相手: {opponent_name}\n「cooperate」か「betray」を選べ。"
    options: [cooperate, betray]
    output: { action: string, inner_thought: string }

  - type: score_calc
    logic: prisoners_dilemma

  - type: summarize
    template: "{agent1}({action1}) vs {agent2}({action2}) → {score1}, {score2}"
```

### 4.4 シナリオ作成ワークフロー（MVP）

```
方法1: プリセットを選ぶ
  アプリ内のサンプルシナリオから選択 → すぐ実行

方法2: 外部LLMで生成
  Claude/ChatGPT等に「Pastura用のシナリオを作って」と依頼
    ↓
  シナリオ定義（YAML）が生成される
    ↓
  アプリにコピペ（テキストフィールド or ファイル共有）
    ↓
  バリデーション → 実行

方法3: 手動でYAMLを書く
  技術者向け。直接YAMLを記述
```

### 4.5 シナリオ生成プロンプト（ユーザーが外部LLMに渡す用）

アプリ内またはドキュメントで以下のようなプロンプトテンプレートを提供:

```
あなたはPasturaというAIマルチエージェントシミュレータのシナリオ設計者です。

以下のフェーズタイプを組み合わせてシナリオを定義してください:
- speak_all: 全員が発言
- speak_each: 1人ずつ順番に発言
- vote: 全員が投票
- choose: 選択肢から選ぶ（options必須）
- score_calc: スコア計算（logic名を指定）
- assign: 情報配布
- eliminate: 最多得票者を脱落
- summarize: 結果要約

ユーザーのリクエスト: [ここにユーザーの自然言語が入る]

YAML形式で出力してください。
```

### 4.6 エンジン処理フロー

```
シナリオ定義（YAML）読み込み
  ↓
バリデーション（フェーズタイプ存在確認、必須フィールドチェック）
  ↓
ラウンドループ (1..rounds)
  ↓
  フェーズループ（phases配列を順に処理）
    ↓
    LLMフェーズ → プロンプト構築 → 推論 → JSON解析 → DB保存
    コードフェーズ → 即座に処理 → DB保存
    ↓
    UIに結果を反映
  ↓
シミュレーション完了
```

### 4.7 設計上の制約

- フェーズは定義順に直列実行（動的分岐はPhase 2の `conditional` で対応）
- LLMに確定的な判断をさせない（スコア計算・勝敗判定は全てコード）
- 進行AIは置かない（ルール破綻リスクの回避）
- score_calc のロジックはアプリに組み込み済みのものから選択（MVP）

## 5. エージェント設計

### 5.1 プロンプト構造

```
[システムプロンプト]           ≈ 200 tokens
  - シナリオのcontext
  - エージェントのペルソナ
  - JSON出力フォーマット指示

[現在の状態]                  ≈ 200〜500 tokens
  - スコアボード、履歴などフェーズ変数を展開

[直近の会話ログ]              ≈ 500〜1,500 tokens
  - 最新5〜10ターン

[フェーズ固有プロンプト]       ≈ 50〜100 tokens
  - YAML定義のpromptフィールドを変数展開

合計: 約 950〜2,300 tokens/回
```

### 5.2 出力安定化（E2B実機テストから得た知見）

| 問題 | 原因 | 対策 |
|------|------|------|
| JSON途中切れ | max_tokensに到達 | max_tokens=1000、出力を簡潔に指示 |
| 無言出力（「...」） | フォーマット理解不足 | few-shot例の追加、空欄禁止を明示 |
| コードブロック囲み | E2Bの癖 | パーサーで```を自動除去 |
| thinkingタグ混入 | Gemma 4の仕様 | パーサーでthinkingタグを除去 |
| 改行入りJSON | 指示不足 | 「JSONは1行で」を明示 |
| パース失敗の残存 | 上記の複合 | 最大2回のリトライ機構 |

### 5.3 推論速度（実測値）

M1 MacBook Air / Ollama / Gemma 4 E2B:
- 1推論: 約15〜20秒（プリフィル + デコード）
- 5体 × 1フェーズ: 約1.5分
- 2ラウンド（囚人のジレンマ）: 約6.5分

iPhone GPU（理論値）: 上記の 1/2〜1/3 程度まで高速化の見込み。

## 9. プリセットシナリオ

プリセットは「サンプル実験」の位置づけ。ユーザーが「こういうものが作れるのか」と理解するためのリファレンス。

### 選定基準（実験プラットフォーム版）

- **フェーズタイプの多様性**: 各プリセットが異なるフェーズタイプを使い、エンジンの能力を示す
- **改変しやすさ**: パラメータを少し変えるだけで違う実験になる
- **結果の観察しやすさ**: スコアや投票結果など、定量的な結果が出る

### プリセット案

| # | シナリオ | 使用フェーズ | 示すこと |
|---|---------|-------------|---------|
| 1 | 囚人のジレンマ | speak_all, choose, score_calc | 戦略ゲーム。ペルソナ変更で結果が変わる |
| 2 | ボケて | assign, speak_all, vote, score_calc | 創造タスク。お題やスタイル変更で実験 |
| 3 | ワードウルフ or 投票サバイバル | assign, speak_each, vote, eliminate | 脱落ゲーム。人数やルール変更で実験 |

※ プロトタイプテストで3つ目を確定する。

## 10. 既知のリスクと対策

### 10.1 技術リスク

| リスク | 影響度 | 対策 |
|--------|--------|------|
| E2Bの出力品質 | 中（ピボットにより影響度低下） | 実験ツールとして「品質の低さ自体が観察対象」と位置づけ |
| JSON出力の不安定さ | 中 | リトライ機構、パーサー堅牢化（実証済み） |
| ユーザー定義YAMLの不正 | 高 | バリデーション + わかりやすいエラーメッセージ |
| 発熱・バッテリー | 中 | 推論ペース制御 |
| LiteRT-LM iOS SDK | 中 | AI Edge Gallery実装を参考 |

### 10.2 プロダクトリスク

| リスク | 影響度 | 対策 |
|--------|--------|------|
| YAML記述のハードルが高い | 高 | シナリオ生成プロンプトの提供、プリセットの改変から始める導線 |
| ターゲットが狭すぎる | 中 | まず技術者層で検証。Phase 2でビジュアルエディタを作り一般層へ拡大 |
| App Store審査 | 中 | NGワードフィルタ、TestFlight配布前にガイドライン確認 |
| 「で、何が面白いの？」問題 | 高 | プリセットの初回体験を工夫。結果のSNS共有を促すUI |
