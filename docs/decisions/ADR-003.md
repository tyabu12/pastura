# ADR-003: Background Simulation Continuation via iOS 26 BGContinuedProcessingTask

> **Status:** Accepted
> **Date:** 2026-04-14
> **Context:** Phase 1 MVP ŌĆö ADR-002 ┬¦7 documented that backgrounding the app pauses the
> simulation. iOS 26 introduces `BGContinuedProcessingTask`, enabling CPU work to continue
> while the app is backgrounded.

## Summary

Add opt-in background simulation continuation for iOS 26+ using `BGContinuedProcessingTask`.
When the user enables the toggle in SimulationView and backgrounds the app, the model
is reloaded from GPU to CPU mode so inference can continue under Background Task API limits.
The feature is gated on iOS 26+ availability and `LlamaCppService` (the only backend that
supports hot reload).

---

## 1. Context: What Changed and Why Now

### ADR-002 ┬¦7 (Original Lifecycle)

ADR-002 ┬¦7 established that `scenePhase == .background` triggers a pause ŌĆö Metal GPU buffers
may be invalidated by the OS when the app is suspended, and resuming with stale buffers
causes `EXC_BAD_ACCESS` or corrupt output. Background inference was not attempted.

### iOS 26: BGContinuedProcessingTask

iOS 26 introduces `BGContinuedProcessingTask` (part of BackgroundTasks framework) ŌĆö a
user-initiated background task type designed for long CPU-bound work that the user started
and expects to continue while the app is backgrounded (e.g., large file processing, AI inference).

Key properties:
- **User-initiated requirement**: the system requires that the task is triggered directly by
  a user action, not automatically. A toggle button tap in SimulationView satisfies this.
- **CPU work only (iPhone)**: background GPU access is iPad-only and requires a separate
  entitlement (`com.apple.developer.background-tasks.continued-processing.gpu`). For simplicity
  and device parity across iPhone and iPad, we chose CPU-only background inference for MVP.
- **Expiration handler**: the system provides an `expirationHandler` closure when it needs the
  task to end; the app must stop work and call `setTaskCompleted(success:)` promptly.

### Why CPU Inference in Background?

Switching from GPU to CPU for background execution is necessary because:

1. Metal GPU contexts are not guaranteed to be available when the app is backgrounded.
2. Background GPU entitlement is iPad-only and not yet tested on device.
3. CPU inference, while slower, is fully functional and deterministic under BG QoS.
4. `LlamaCppService` supports hot reload via `unloadModel()` + `loadModel()`, making
   the GPUŌåöCPU switch safe as long as no inference is in-flight.

---

## 2. Decision

Implement opt-in background simulation continuation with the following constraints:

- **Default OFF**: the toggle is disabled by default each simulation session.
  Per-simulation reset avoids surprise background CPU usage.
- **iOS 26+ only**: gated via `if #available(iOS 26, *)`. Falls back to pause-on-background
  for iOS 17ŌĆō25 (original ADR-002 ┬¦7 behaviour unchanged).
- **LlamaCppService only**: only `LlamaCppService` exposes `reloadModel(gpuAcceleration:)`.
  Other backends (Ollama, Mock) are always remote or in-memory ŌĆö reload is meaningless.
- **CPU-only for background**: `n_gpu_layers: -1` (full GPU) in foreground;
  `n_gpu_layers: 0` (CPU only) in background. No background GPU entitlement required.
- **Foreground restore**: when the app returns to foreground, the model is reloaded back to
  GPU mode. Simulation pauses briefly during the reload, then resumes automatically.

## 3. Architecture Components

### 3.1 `GPUAcceleration` Enum (LLM/)

```swift
/// Controls Metal GPU layer offloading for LlamaCppService.
public enum GPUAcceleration: Sendable {
    case full   // n_gpu_layers = -1 (all layers on GPU)
    case none   // n_gpu_layers = 0 (CPU only)
}
```

Maps directly to `llama_model_params.n_gpu_layers`. Adding the enum keeps call sites
readable and avoids magic numbers at the reload site.

### 3.2 `LlamaCppService.reloadModel(gpuAcceleration:)` (LLM/)

```swift
func reloadModel(gpuAcceleration: GPUAcceleration) async throws
```

Sequential unload + reload with the new acceleration setting. Protected by
`generatingGuard` ŌĆö a `precondition` that crashes if called during an in-flight
`generate()` call. Callers are responsible for pausing the simulation and confirming
the `.simulationPaused` event before invoking this method.

### 3.3 `BackgroundSimulationManager` (App/)

Wraps `BGTaskScheduler` and `BGContinuedProcessingTask`:

- `requestContinuation()` ŌĆö submits a `BGContinuedProcessingTask` request immediately
  after the toggle is enabled and the app backgrounds. Returns `true` if the system
  grants the task.
- `cancelContinuation()` ŌĆö cancels any pending or active task when the simulation ends
  or the user disables the toggle.
- Expiration callback ŌĆö fires the `onExpired` closure passed in by `SimulationViewModel`
  so ViewModel can respond per the state machine in ┬¦5.

Placed in `App/` to satisfy the dependency rule: BackgroundTasks framework is an
App-layer concern; `LLM/` and `Engine/` must not import it.

### 3.4 `SimulationViewModel` (App/)

Three new entry points:

| Method | Trigger | Action |
|--------|---------|--------|
| `enableBackgroundContinuation` | Toggle tapped ON in SimulationView | Sets flag; registers task identifier with BGTaskScheduler |
| `handleScenePhaseBackground()` | `scenePhase == .background` | Calls `pauseAndAwaitConfirmation()`, then `reloadModel(.none)`, then submits BG task |
| `handleScenePhaseForeground()` | `scenePhase == .active` | Calls `reloadModel(.full)`, then resumes simulation |

### 3.5 `SimulationView` Toggle (Views/)

A toggle button in the simulation controls bar, rendered only when
`#available(iOS 26, *)` AND the active `LLMService` is `LlamaCppService`.
Label: "Continue in Background". Default state: off.

The toggle is hidden entirely when unavailable to avoid confusing users on
older OS versions or when using Ollama/Mock backends.

## 4. Safe Reload Pattern (Critical)

`LlamaCppService.generatingGuard` is a `precondition` ŌĆö it crashes with a clear message
if `reloadModel` is invoked while a `generate()` call is active. This is intentional:
a silent race would produce corrupt output or a crash without a useful stack trace.

The safe reload sequence is:

```
scenePhase = .background
        Ōöé
        Ō¢╝
SimulationViewModel.handleScenePhaseBackground()
        Ōöé
        Ō¢╝
SimulationRunner.isPaused = true          ŌåÉ sets flag
        Ōöé
        Ōöé  (current generate() completes naturally)
        Ōöé
        Ō¢╝
SimulationRunner.checkPaused()            ŌåÉ checked at ROUND boundary
        Ōöé                                    AND PHASE boundary (this PR)
        Ō¢╝
emitter(.simulationPaused(...))
        Ōöé
        Ō¢╝
SimulationViewModel.pauseAndAwaitConfirmation()
        Ōöé                                    ŌåÉ CheckedContinuation resumes here
        Ō¢╝
LlamaCppService.reloadModel(.none)        ŌåÉ safe: no generate() in-flight
        Ōöé
        Ō¢╝
BackgroundSimulationManager.requestContinuation()
        Ōöé
        Ō¢╝
SimulationRunner resumes on CPU
```

### `SimulationRunner.checkPaused()` Call Sites

Previously only called at round boundaries. This PR adds a call at each **phase boundary**
within a round. This ensures the pause-and-reload can happen between individual LLM calls
regardless of how many phases remain in the current round.

### `pauseAndAwaitConfirmation()` Implementation

```swift
// SimulationViewModel
func pauseAndAwaitConfirmation() async {
    simulationRunner.isPaused = true
    await withCheckedContinuation { continuation in
        pauseContinuation = continuation
    }
    // Resumed when .simulationPaused event arrives in the event loop.
}
```

Zero polling ŌĆö the continuation is resumed exactly once when the Engine emits
`.simulationPaused`, guaranteeing the reload starts only after inference is idle.

## 5. State Machine for BG Task Expiration

When the OS fires `expirationHandler`, the app must stop promptly. Different responses
are needed depending on what the simulation is doing at that moment:

| Stage | Action on Expiration |
|-------|---------------------|
| Before activation | No-op (task was never started by the system; nothing to clean up) |
| During pause-await | Wake the `CheckedContinuation` immediately; leave simulation paused; call `setTaskCompleted(success: false)` |
| During model reload | Let the current unload+load finish (fast: ~3-8s, within expiration window); then call `setTaskCompleted(success: false)` |
| Running on CPU | Set `isPaused = true` on `SimulationRunner`; call `setTaskCompleted(success: false)` ŌĆö simulation resumes when user returns to foreground |
| After user returns to foreground | No-op (task was already completed when foreground restore began) |

The `success: false` flag in all expiration cases signals to the system that the work
did not finish, which is accurate ŌĆö the simulation is paused, not complete.

## 6. Performance Characteristics

### GPUŌåöCPU Reload Time

| Phase | Estimated Duration |
|-------|--------------------|
| `unloadModel()` | ~0.5-1s (free Metal buffers + weights) |
| `loadModel()` CPU | ~2-7s (re-read model file from disk, map weights to RAM) |
| **Total GPUŌåÆCPU** | **~3-8s** |
| **Total CPUŌåÆGPU** | **~3-8s** (same path, different n_gpu_layers) |

The 3-8s reload window is well within the `BGContinuedProcessingTask` expiration grace period
(which is not publicly documented but empirically observed at 30s+ for user-initiated tasks).

### CPU Inference Speed in Background

| Mode | Approx. Speed |
|------|--------------|
| Foreground GPU (A17/A18) | ~20-40 tok/s |
| Background CPU | ~2-8 tok/s (~0.1-0.2├Ś GPU) |

The additional slowdown versus a typical CPU-foreground number is due to Background QoS
throttling: the OS assigns `.utility` or `.background` QoS to BG task threads, reducing
CPU scheduling priority.

**User expectation**: simulation continues, but noticeably slower. A progress indicator
showing "Running in background (slower)" is shown in the notification/Dynamic Island (if supported).

### Memory: GPU vs CPU

| Mode | Approx. RAM |
|------|------------|
| Foreground GPU (Q4_K_M, Metal buffers) | ~4.5-5.0 GB |
| Background CPU (Q4_K_M, no Metal) | ~3.5-4.5 GB |

CPU mode uses slightly less RAM (no Metal buffer allocation), reducing jetsam risk during
background execution. However, the app is still at risk if the OS needs the memory urgently.

### Thermal Throttling

The existing `throttleIfOverheating()` mechanism (ADR-002 ┬¦5) continues to apply in background.
At `.serious` or `.critical` thermal state, a 200ms pause is inserted between inferences.
CPU inference generates less heat than GPU, reducing the likelihood of sustained thermal
throttling during background execution.

## 7. Trade-offs Considered

### iPad GPU Background

**Status: Deferred**

iPad supports background GPU access via the
`com.apple.developer.background-tasks.continued-processing.gpu` entitlement (iOS 26+, iPad only).
This would allow ~20-40 tok/s background inference on iPad instead of ~2-8 tok/s CPU.

Deferred because:
- Requires an entitlement not yet requested or approved
- No iPad test device available for validation
- Complicates the implementation: GPU/CPU branch per device + per scene phase
- Device parity story is cleaner without it for MVP

Tracked as future work (┬¦8).

### Automatic (Always-On) Background Continuation

**Rejected**

Making background continuation automatic (always enabled, no toggle) was rejected:
- Battery impact is significant ŌĆö CPU inference at BG QoS still consumes meaningful power
- Background tasks require user initiation (OS enforces this)
- Users who do not want background AI activity should not be surprised by it

The toggle must be explicitly enabled before backgrounding.

### Per-App-Session vs Per-Simulation Toggle State

**Decision: Per-simulation (resets on each simulation start)**

Resetting the toggle for each simulation start is safer: users who forget they enabled it
for a previous run are not surprised. A persistent user preference (remember last state)
is deferred to Phase 2 (┬¦8).

## 8. Future Work

- **iPad GPU background**: request `com.apple.developer.background-tasks.continued-processing.gpu`
  entitlement; add device-type check; test on physical iPad Pro M4+. Update ADR-003 when implemented.
- **Persistent toggle preference**: remember last background continuation state across app launches
  via `UserDefaults`. Reset on first launch of a new app version.
- **Thermal-aware dynamic throttling**: in background, monitor thermal state and adjust inter-inference
  delay dynamically (e.g., 500ms at `.serious`, yield task at `.critical`).
- **Checkpoint save on backgrounding**: serialize `SimulationState` to disk before entering background
  so a jetsam kill during suspension does not lose round progress. Assess complexity before committing.
- **Dynamic Island / Live Activity**: show simulation progress (current round / total) while backgrounded.
  Requires ActivityKit entitlement (separate PR).
