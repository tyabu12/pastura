> **Status:** Accepted
> **Date:** 2026-04-06
> **Context:** Pastura Phase 0 completed with conditional Go. Entering Phase 1 (MVP development).

## Summary

This ADR captures all architecture decisions made during Phase 1 design for the Pastura MVP.
Each decision is recorded with context, options considered, and rationale.

## 1. Project Structure: Directory-based, No SPM Module Split

### Context

The MVP has an estimated 40–50 Swift files across 5 logical layers (Models, LLM, Engine, Data, Views).
SPM multi-target splits enforce dependency direction at compile time but add overhead for Claude Code workflows.

### Decision

Use directory-based separation with strict dependency rules documented in CLAUDE.md.
No SPM module split for MVP.

### Rationale

- At 40–50 files, build time difference is negligible (clean build ~30–60s on M1 MacBook Air 16GB)
- Directory structure mirrors future module boundaries — migration cost is ~half a day when needed
- Claude Code instructions stay simple ("create file at Engine/Phases/X.swift" vs managing Package.swift targets)
- YAGNI: module split benefits (build time, compile-time dependency enforcement) kick in at 100+ files

### Migration Path

When file count exceeds ~100 or dependency violations become frequent:
1. Add targets to Package.swift matching existing directory structure
2. Add `import` statements across module boundaries
3. Change `internal` → `public` on cross-module types (protocols and Models/ types are already `public`)

## 2. YAML Decoding: Manual Mapping via Yams.load()

### Context

Scenario definitions have a dynamic structure — the `phases` array contains elements with different
fields depending on `type`, and `output` has user-defined field names. Two decoding approaches exist.

### Options

| Option | Pros | Cons |
|--------|------|------|
| **A: Yams.load() → [String: Any] → manual mapping** | Handles YAML format variations gracefully; mirrors Python prototype; straightforward debugging | Less type safety at parse time; more boilerplate |
| **B: YAMLDecoder + Codable** | Compile-time type safety; less boilerplate | Fragile against YAML format variations (inline vs block style); complex custom `init(from:)` for Phase union type |
| **C: Hybrid** | Fixed fields via Codable, phases via manual mapping | Two parsing paths to maintain |

### Decision

Option A: Manual mapping via `Yams.load(yaml:)`.

### Rationale

- The Python prototype uses the same approach (`yaml.safe_load` → dict) — proven to work with E2B-generated and user-authored YAML
- YAML from external LLMs (Claude, ChatGPT) includes format variations (code fences, inline vs block maps) that Codable handles poorly
- Validation logic (field presence, type checking, range limits) is explicit and easy to test
- Phase 2's `conditional` and `event_inject` phase types will add more dynamic structure — manual mapping scales better

## 3. SQLite Library: GRDB v7.10.0

### Context

The app needs local persistence for scenarios, simulation state (pause/resume), and turn history.

### Options

| Option | Pros | Cons |
|--------|------|------|
| **GRDB** | Strong Codable integration; migration system; SwiftUI observation support (GRDBQuery); Swift 6 ready; cross-platform (Android/Linux/Windows) | Learning curve for query interface |
| **SQLite.swift** | Lightweight; type-safe query builder | Weaker Codable story; no built-in migration |
| **SwiftData** | Apple-native; tight SwiftUI integration | iOS 17+ only (OK); limited migration control; opaque debugging; overkill for simple schema |
| **Core Data** | Mature; Apple-supported | Heavy; complex for simple schemas |

### Decision

GRDB v7.10.0.

### Rationale

- Codable integration maps directly to our `SimulationState` serialization needs
- DatabaseMigrator provides controlled schema evolution for future phases
- Cross-platform support aligns with potential KMP strategy (though Data layer would be platform-specific regardless)
- Battle-tested in production apps; active maintenance

## 4. Database Schema: 3 Tables, No Agents Table

### Context

The MVP spec (v0.3.0) defined 4 tables: scenarios, simulations, agents, turns.
Agent state (scores, elimination) is part of SimulationState which is already serialized to JSON.

### Decision

3 tables: `scenarios`, `simulations`, `turns`. No `agents` table.
Agent state is stored in `simulations.stateJSON` as part of the Codable `SimulationState`.

### Rationale

- Agent list is always available from the scenario's YAML definition (personas array)
- Scores and elimination status are already fields of SimulationState
- Avoids FK joins for common queries (result display reads stateJSON + turns)
- If Phase 2 needs per-agent analytics (score history, etc.), an agents table can be added via migration

### Trade-off

Cannot directly query agent data via SQL (e.g., "find all simulations where agent X scored > 10").
Acceptable for MVP where result display deserializes stateJSON in application code.

## 5. SimulationRunner Output: AsyncStream\<SimulationEvent\>

### Context

The simulation engine produces events incrementally (agent outputs, phase completions, round transitions).
The UI needs to display these in real-time.

### Options

| Option | Pros | Cons |
|--------|------|------|
| **AsyncStream** | Native Swift Concurrency; `for await` in SwiftUI `task`; backpressure; Task cancellation | No built-in pause; requires custom suspension mechanism |
| **@Observable ViewModel** | Direct mutation; simple | Tight coupling between Engine and UI; hard to test |
| **Combine Publisher** | Mature; operators | Adding Combine dependency when Swift Concurrency suffices |

### Decision

AsyncStream\<SimulationEvent\> with isPaused flag and CheckedContinuation for pause.

### Rationale

- Natural fit with SwiftUI's `task { for await event in ... }` pattern
- Engine remains UI-agnostic — it just yields events
- Cancellation via Task.cancel() provides clean shutdown
- No performance concern: bottleneck is LLM inference (seconds per call), not event dispatch (nanoseconds)
- No lock-in: can bridge to @Observable or Combine trivially if needed

### Pause Implementation

```swift
// Inside SimulationRunner.checkPaused():
guard pauseState.withLock({ $0.isPaused }) else { return false }
emitter(.simulationPaused(round: round, phaseIndex: 0))
await withTaskCancellationHandler {
    await withCheckedContinuation { continuation in
        // Store continuation; isPaused setter resumes it on unpause.
    }
} onCancel: { /* resume continuation if stored */ }
```

Signal-based: zero CPU during pause, emits `simulationPaused` exactly once per cycle.
Pause during LLM inference naturally waits for the current inference to complete
before checking the flag.

## 6. PhaseHandler Event Notification: Emitter Closure

### Context

PhaseHandlers need to report events (agent outputs, phase start/end) back to SimulationRunner,
which feeds them into the AsyncStream.

### Decision

Emitter closure: `(SimulationEvent) -> Void` passed to each PhaseHandler.

### Rationale

- AsyncStream's `continuation.yield()` can be passed directly as the emitter
- Tests can pass a closure that appends to an array for assertion
- Simpler than delegate pattern (no protocol to define, no weak references)
- Each PhaseHandler calls `emitter(.agentOutput(...))` inline — clear and debuggable

## 7. LLM Backend Strategy: Protocol Abstraction with OllamaService for Development

### Context

LiteRT-LM Swift SDK is not yet released ("Swift APIs coming soon" as of 2026-04).
The AI Edge Gallery iOS source code is not public (Issue #420 open since 2026-01).
Development cannot wait for SDK release.

### Decision

1. Define `LLMService` protocol with `loadModel()`, `unloadModel()`, `isModelLoaded`, `generate(system:user:)`
2. Develop with `OllamaService` (connects to Ollama on Mac via OpenAI-compatible API)
3. Test with `MockLLMService` (returns pre-defined responses)
4. Production integration priority: Swift SDK → C API wrapper → MediaPipe (deprecated)

> **Update (2026-04-08):** llama.cpp was selected as the interim TestFlight backend
> due to Metal GPU support on iOS. See [ADR-002](ADR-002.md) for details and migration plan.

### Rationale

- Protocol abstraction means Engine and UI code have zero dependency on any specific LLM backend
- OllamaService works on both Simulator and physical devices (when on same network as Mac)
- MockLLMService enables deterministic, fast Engine tests
- When Swift SDK releases, only one file needs to be written (swap the backend implementation)

### Model Lifecycle

- Load at simulation start (not app launch) to conserve memory
- Unload after simulation completes
- Immediate unload on `didReceiveMemoryWarning`
- Timing is adjustable — only call site in App/ layer changes, not the protocol

## 8. YAML Validation Limits

### Context

Phase 0 testing showed that 5 agents × 9 speak_each rounds took 93 minutes.
Unrestricted parameters could lead to simulations running for hours.

### Decision

| Parameter          | Limit  | Behavior              |
|--------------------|--------|-----------------------|
| Agent count        | ≤ 10   | Validation error      |
| Round count        | ≤ 30   | Validation error      |
| Estimated inferences | > 50 | Warning (UI display)  |
| Estimated inferences | > 100| Validation error      |

### Rationale

- 10 agents × 30 rounds is generous for experimentation while preventing runaway simulations
- Inference estimation formula provides actionable feedback ("~X minutes estimated")
- Limits are adjustable via constants — can be relaxed after real-device performance testing
- Phase 0 data: E2B GPU at ~56 tok/s → ~2.7s per inference → 50 inferences ≈ 2.3 min, 100 ≈ 4.5 min

## 9. Minimum iOS Version: 17.0

### Context

Key framework requirements: @Observable (iOS 17+), AsyncStream (iOS 15+), SwiftUI (iOS 13+).

### Decision

iOS 17.0 minimum.

### Rationale

- @Observable macro (iOS 17+) is the recommended state management approach for SwiftUI
- All target devices (iPhone with 8GB+ RAM for LLM) are iPhone 15 or later, which support iOS 17
- iOS 18+ offers no critical features for MVP (mesh gradients, @Previewable are nice-to-have)
- Wider device coverage for TestFlight testing

## 10. Content Filter Placement: Between Engine and UI

### Context

NGWordFilter is needed for App Store compliance. Debug mode needs access to raw LLM output.

### Decision

Filter is applied in the App/ViewModel layer, between Engine output and UI display.
Engine returns raw, unfiltered TurnOutput.
DB stores raw output in TurnRecord.rawOutput.

### Rationale

- Engine stays pure — no policy decisions about content
- Filter logic is testable independently
- App Store review sees filtered output in all modes including debug
- Raw output preserved in DB (TurnRecord.rawOutput) for developer inspection via dedicated debug UI
- Clear separation: Engine → raw data → ContentFilter → displayed data

## 11. score_calc Extensibility: 3 Built-in Logics, No Custom for MVP

### Context

Users creating custom scenarios may need scoring logic beyond the 3 built-in options.

### Decision

MVP includes exactly 3 scoring logics: `prisoners_dilemma`, `vote_tally`, `wordwolf_judge`.
Custom logic definition is Phase 2 scope.

### Rationale

- `vote_tally` is generic enough to cover many user scenarios
- Custom logic requires designing a safe expression language or scripting sandbox — significant scope
- Phase 0 testing showed the 3 preset scenarios cover diverse phase type combinations
- YAGNI: validate demand via TestFlight feedback before investing

## 12. Platform Strategy: iOS-Only MVP, Language-Agnostic Design

### Context

Swift + SwiftUI limits the app to Apple platforms. Future Android/PC demand is anticipated
(especially for YAML editing on PC).

### Decision

MVP is iOS-only. Design Engine logic and scenario format to be language-agnostic.

### Key Points

- YAML scenario definitions work on any platform
- Engine algorithms (phase dispatch, score calculation, template expansion) are portable logic
- LiteRT-LM has official SDKs for Android (Kotlin), JVM Desktop, Python, and Web (JS)
- Python prototype already serves as a PC reference implementation
- KMP (Kotlin Multiplatform) is a viable future path for shared Engine code

### Migration Cost Assessment

If porting to Android/KMP:
- Models + Engine logic: rewrite in Kotlin (algorithm port, not redesign)
- LLM layer: swap to `litertlm-android` SDK (same concepts, different API)
- Data layer: GRDB has Android support, or use Room
- UI: full rewrite in Compose (expected)
- YAML scenarios + presets: zero cost (shared as-is)
